# citemap.json Specification v3.1 Addendum

> Two backward-compatible additions that bridge citemap.json to the Schema.org content graph

**Status:** Released
**Version:** 3.1
**Date:** April 2026
**Authors:** Brian Pofahl
**License:** CC BY 4.0
**Schema:** https://citemaps.org/schema/v3.1/citemap.schema.json
**Backward Compatibility:** Every valid v2.0 and v3.0 citemap validates against v3.1

---

## Overview

CiteMap v3.1 adds two features on top of the [v3.0 specification](./v3.0.md). Both additions are optional and backward-compatible — no v2.0 or v3.0 fields are removed, renamed, or changed in meaning. A v3.0 file passes v3.1 validation without modification.

v3.1 addresses a single coordination gap: **the relationship between a citemap and a site's Schema.org `@graph`**. Sites running Yoast SEO, Rank Math, Schema Pro, or any custom JSON-LD generator already publish a rich content graph — Articles, Products, Events, Authors, Organizations, and the relationships between them. Today, AI systems and crawlers have to discover and reconcile the two surfaces independently. v3.1 lets citemap.json declare the pointer and offer a join key.

The two additions are:

1. **Schema Graph Reference** — A root-level pointer to the site's Schema.org `@graph` endpoint
2. **Schema Entity IDs** — An optional `schemaId` cross-reference on any v3.0 Entity ID, resolving to a JSON-LD `@id` in the graph

Together they turn citemap.json into a *discovery root*: an AI crawler pulls one file, gets canonical identity + citation contract + trust stack, and is told exactly where to find the content graph and how to stitch the two together.

### Design Principle: Don't Absorb, Bridge

citemap.json is not a Schema.org replacement and v3.1 does not try to become one. The two formats serve different, complementary jobs:

| | citemap.json | Schema.org `@graph` |
|---|---|---|
| **Answers the question** | Who is this entity, and how should I cite it? | What exists on this site, and how is it connected? |
| **Authoritative for** | Identity, citation contract, verifiable claims, trust signals | Content relationships, page-level metadata, product catalogs, article authorship |
| **Payload shape** | One canonical file per entity (~10–100 KB) | Many pages, each with embedded JSON-LD; aggregated on demand |
| **Typical producer** | The entity directly, via CiteMaps.ai or hand-authored | Yoast SEO, Rank Math, Schema Pro, custom plugins, framework adapters |

v3.1 respects this boundary. It adds a pointer and a join key — not a second content graph.

## 1. Schema Graph Reference

### Problem

A site running Yoast SEO's Schema Aggregation endpoint (or equivalent) already publishes a full Schema.org `@graph` at a known URL — for example `https://example.com/wp-json/yoast/v1/get_head?url=/`. An AI crawler that has only the citemap has no machine-readable way to know that graph exists, where to fetch it, or which generator produced it. The crawler either duplicates discovery work (re-crawling every page for embedded JSON-LD) or misses the graph entirely.

### Solution

A new root-level `schemaGraph` object that points to the site's aggregated `@graph` and describes its shape.

### Schema

```json
{
  "schemaGraph": {
    "url": "https://example.com/wp-json/yoast/v1/get_head?url=/",
    "format": "schema.org-jsonld",
    "generator": "yoast",
    "generatorVersion": "22.4",
    "lastUpdated": "2026-04-20",
    "coverage": "site",
    "entityCount": 142
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | string (URI) | **Yes** | Absolute URL where the aggregated Schema.org `@graph` can be fetched. MUST return `application/ld+json` or `application/json` with a top-level `@graph` array. |
| `format` | enum | No | The serialization format. Current values: `schema.org-jsonld` (default), `schema.org-jsonld-html` (embedded in HTML via `<script type="application/ld+json">`), `json-ld-other`. |
| `generator` | enum \| string | No | The tool that produces the graph. Known values: `yoast`, `rank-math`, `schema-pro`, `all-in-one-seo`, `wpseo-schema`, `next-seo`, `nuxt-schema-org`, `custom`. Free-form strings are allowed for unknown generators. |
| `generatorVersion` | string | No | Version of the generator. Useful when tooling behavior varies across versions. |
| `lastUpdated` | string (ISO 8601 date) | No | When the graph was last regenerated. Most aggregation endpoints serve fresh graphs on every request — this field is for graphs that are cached or built statically. |
| `coverage` | enum | No | Scope of the graph. Values: `site` (full-site aggregation), `entity` (only entities related to this citemap), `home` (homepage @graph only). Default: `site`. |
| `entityCount` | integer | No | Approximate count of top-level graph entities. Helps validators and AI systems decide whether to fetch the full graph or paginate. |

### Inline Fallback

For sites without a native aggregation endpoint (static-site generators, brochure sites, or environments where a crawler cannot resolve an authenticated URL), v3.1 allows an optional inline `graph` array:

```json
{
  "schemaGraph": {
    "format": "schema.org-jsonld",
    "generator": "custom",
    "lastUpdated": "2026-04-20",
    "graph": [
      {
        "@context": "https://schema.org",
        "@type": "Organization",
        "@id": "https://example.com/#organization",
        "name": "Example Inc.",
        "url": "https://example.com"
      }
    ]
  }
}
```

Inline graphs are **discouraged by default**. Pointers stay fresh and let the generator remain authoritative; inline graphs create a second source of truth that drifts. Producers SHOULD prefer `url` and fall back to `graph` only when an aggregation endpoint is unavailable.

If both `url` and `graph` are present, the `url` is authoritative and consumers SHOULD prefer it. The inline `graph` acts as a cache/fallback.

### Guidance for AI Systems

When a citemap includes `schemaGraph.url`, AI systems and crawlers SHOULD:

1. Fetch the citemap first (identity + citation contract + trust signals).
2. If the task requires content discovery or relationship data, fetch the referenced `@graph`.
3. Apply the citemap's `citationContract.preferredName` as an override when any Schema.org entity in the graph names the same entity under a different label.
4. Treat fields present in both sources as consistent; if they conflict, the citemap is authoritative for identity fields (`brand.name`, `citationContract.*`, `verifiedClaims[]`) and the `@graph` is authoritative for content fields (article authorship, product prices, event times).

### Validator Behavior

The `@citemap/validator` SHOULD:

- Resolve `schemaGraph.url` with a HEAD or bounded GET request.
- Emit an **info** signal if the URL is unreachable, returns non-JSON, or lacks a top-level `@graph` array. This is not a validation failure — the network state of an external endpoint is not citemap-fatal.
- Emit a **warning** if `generator` is `custom` or absent and `graph` is also absent. In that case the reference points nowhere the validator can verify.
- If `graph` is inlined, validate that each element has `@context`, `@type`, and `@id`.

## 2. Schema Entity IDs

### Problem

v3.0 introduced stable Entity IDs in `type:slug` format (e.g., `service:craft-beer-selection`, `person:betsy-weedman`) on 24 nested object types. Schema.org uses a different identifier pattern — fragment-based URI `@id` values (e.g., `https://3rdstreetbeverage.com/#/schema/person/betsy-weedman`). There is currently no way to assert that the two refer to the same entity. An AI that consumes both a citemap and the site's `@graph` cannot confidently join them.

### Solution

An optional `schemaId` field on any v3.0 Entity ID, holding the corresponding Schema.org `@id` URI.

### Schema

```json
{
  "people": [
    {
      "id": "person:betsy-weedman",
      "schemaId": "https://3rdstreetbeverage.com/#/schema/person/betsy-weedman",
      "fullName": "Betsy Weedman",
      "role": "Co-Owner"
    }
  ],
  "localBusiness": {
    "services": [
      {
        "id": "service:craft-beer-selection",
        "schemaId": "https://3rdstreetbeverage.com/#/schema/service/craft-beer-selection",
        "name": "Craft Beer Selection"
      }
    ]
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `schemaId` | string (URI) | No | The Schema.org `@id` URI of the corresponding entity in the site's `@graph`. When present on an entity that also has a v3.0 `id`, the two are asserted to refer to the same real-world thing. |

### Supported Entities

`schemaId` is available on every v3.0 entity that supports `id`. See [v3.0 §3](./v3.0.md#3-entity-ids) for the full list. In summary: `answerContent[]`, `people[]`, `awards[]`, `heroProducts[]`, `location`, `services[]`, `signatureContent[]`, `events[]`, `courses[]`, `portfolio.featured[]`, `programs[]`, `officials[]`, `emergencyContacts[]`, `studies[]`, `datasets[]`, `trials[]`, `patents[]`, `trademarks[]`, `canonicalQuotes[]`, `practitioners[]`, `products[]`, `attorneys[]`, `timeline[]`.

### Format

`schemaId` MUST be an absolute URI. The fragment-based convention (`https://example.com/#/schema/<type>/<slug>`) is strongly recommended and matches Yoast's default, but any Schema.org-valid `@id` URI is accepted.

### Guidance for AI Systems

When an entity carries both `id` and `schemaId`:

1. Treat the two as co-referential — any claim the citemap makes about `id` applies to the `@graph` node with the matching `@id`, and vice versa.
2. Use the citemap's fields as authoritative for the identity subset (name, role, `citationContract` if scoped to the entity, `verifiedClaims` references).
3. Use the `@graph` node as authoritative for content-level data (article authorship, structured properties Schema.org specifies that citemap does not).

### Validator Behavior

The `@citemap/validator` SHOULD:

- Validate that `schemaId` is a syntactically valid URI.
- If `schemaGraph` is present (either `url` or inline `graph`), attempt to resolve each `schemaId` against the graph. Emit an **info** signal per unresolved `schemaId` — this is useful diagnostic output but not a failure (the graph may be paginated, rate-limited, or fetched asynchronously).
- Count resolved `schemaId` cross-references for the Graph-Linked signal (see below).

## 3. Graph-Linked Trust Signal

v3.1 does not introduce a new Level. The three-tier Level system defined in v3.0 remains the authoritative maturity path. Instead, v3.1 adds an additive trust signal that validators compute independently of Level.

### Definition

A citemap is **Graph-Linked** when all of the following are true:

1. `schemaGraph.url` is present and resolves to a valid `@graph` (or `graph` is inlined and valid).
2. At least one top-level entity in the graph has an `@id` that matches a `schemaId` in the citemap.
3. The `brand.url` host matches the `schemaGraph.url` host (same origin, ignoring subdomain edge cases flagged separately).

### Validator Output

`@citemap/validator` emits a `graphLinked: true` flag and a `graphLinkDensity` ratio (resolved `schemaId` references / total `schemaId` references) on any citemap meeting the definition. CLI output includes a `◆ Graph-Linked` badge alongside the Level badge.

### Why Not a Level 4?

The Level system is a linear maturity ladder: each tier is a superset of the previous. Graph-Linking is **orthogonal** to Level — a Level 1 citemap on a Yoast-powered site can be Graph-Linked, and a Level 3 citemap on a static HTML site may not be. Treating it as a separate signal preserves the clean mental model of v3.0 while surfacing the capability.

## Migration from v3.0

v3.1 is fully backward-compatible. An existing v3.0 citemap remains valid under v3.1 without modification. To adopt v3.1:

### Step 1: Update version fields

```json
{
  "$schema": "https://citemaps.org/schema/v3.1/citemap.schema.json",
  "citemapVersion": "3.1"
}
```

### Step 2: Add a Schema Graph Reference

Locate your site's Schema.org `@graph` endpoint. For common generators:

| Generator | Typical Endpoint |
|-----------|------------------|
| Yoast SEO | `/wp-json/yoast/v1/get_head?url=/` |
| Rank Math | `/wp-json/rankmath/v1/getHead?url=/` |
| Schema Pro | Embedded in page `<head>`; extract via HTML scraper or custom endpoint |
| next-seo | Site-specific; typically exposed via a Next.js API route |
| Custom | Site-specific |

Add the pointer:

```json
{
  "schemaGraph": {
    "url": "https://yoursite.com/wp-json/yoast/v1/get_head?url=/",
    "generator": "yoast"
  }
}
```

### Step 3: Add Schema Entity IDs to key entities

For each v3.0 entity that also appears in your `@graph` (typically your organization, key people, primary services, and hero products), add the corresponding `schemaId`:

```json
{
  "id": "person:betsy-weedman",
  "schemaId": "https://yoursite.com/#/schema/person/betsy-weedman"
}
```

Prioritize entities that AI systems are most likely to cite — owners, founders, signature services, and featured content.

### Step 4: Validate and check Graph-Linked status

```bash
npx @citemap/cli validate ./citemap.json
npx @citemap/cli diagnose ./citemap.json --check-graph
```

The `--check-graph` flag resolves `schemaGraph.url` and counts cross-resolved `schemaId` references, reporting your Graph-Linked status and link density.

## Design Notes

### Why not embed the full graph by default?

Two reasons. First, freshness — generators like Yoast rebuild the graph on every request from the live database; an inline copy in citemap.json decays the moment a post is published, an author bio changes, or a product is added. Second, boundaries — citemap.json is for identity and citation, not content discovery. Inlining the full graph conflates the two and pressures citemap to grow in ways that duplicate work the Schema.org ecosystem already does well.

The inline `graph` fallback exists because there are legitimate environments (static sites, brochure sites, archived records) where an aggregation endpoint is unavailable. In those cases a bounded inline graph is better than no graph at all.

### Why `schemaId` on the entity rather than a separate mapping table?

A mapping table (`{ "person:betsy-weedman": "https://.../schema/person/betsy-weedman", ... }`) would be compact but brittle: renaming an entity's v3.0 `id` would require updating two places, and consumers would need to dereference the table on every lookup. Keeping `schemaId` alongside `id` on the entity itself is local, co-located, and mirrors how v3.0 treats other per-entity metadata.

### Why not a v3.1 citemapLevel 4?

The v3.0 Level system is a superset ladder — each level includes all the requirements of the previous. Graph-Linking is orthogonal to that: it depends on the site's SEO stack, not on how complete the citemap's own fields are. Forcing it into the Level system would either dilute Level 3 (so a Level 3 citemap on a static site would silently downgrade) or invent a Level 4 that most citemaps structurally can't reach. A separate signal is cleaner.

### Multi-generator sites

A small number of sites publish multiple overlapping `@graph` sources (e.g., Yoast plus a custom overlay). v3.1 currently supports one `schemaGraph` reference. Sites with multiple graphs should either pick the authoritative one or merge them server-side before exposing the pointer. A future revision may support `schemaGraph: [...]` as an array if real-world demand emerges — but the default expectation is one graph per citemap, matching the one-citemap-per-entity model.

## Changelog

### v3.1 — April 2026

**Two backward-compatible additions that bridge citemap.json to the Schema.org content graph.** Root-level `schemaGraph` object points to a site's aggregated JSON-LD `@graph` (Yoast Schema Aggregation, Rank Math, next-seo, or custom), with optional inline fallback for environments without an aggregation endpoint. Entity-level `schemaId` provides a join key between v3.0 Entity IDs (`type:slug`) and Schema.org `@id` URIs, turning citemap.json into the discovery root for both identity and content. A new Graph-Linked trust signal — orthogonal to the v3.0 Level system — surfaces on validators and the CLI when the graph resolves and cross-references land.

Every valid v2.0 and v3.0 citemap passes v3.1 validation without modification.
