# citemap.json v3.2.1 Addendum — Registry Token

> One backward-compatible field addition that lets publishers claim their citemap entry on the citemaps.org public registry.

**Status:** Draft
**Version:** 3.2.1
**Date:** May 2026
**Authors:** Brian Pofahl
**License:** MIT
**Backward Compatibility:** Every valid v2.0, v3.0, v3.1, and v3.2 citemap validates against v3.2.1. The new field is optional and additive; v3.2-or-earlier consumers ignore it.

---

## Overview

v3.2.1 adds one optional string field — `citationContract.registryToken` — that publishers use to prove ownership of their citemap when claiming their entry on the [citemaps.org public registry](https://citemaps.org/registry).

The field is harmless when unclaimed. It is intentionally not a secret — it's a publisher-controlled identifier embedded in a public document. Its security model is *proof of authorship*, not *proof of identity*: anyone who can publish the citemap at the URL can also publish the token in it, which is the same thing as controlling the citemap's deployment surface.

---

## 1. Field Specification

### `citationContract.registryToken`

**Type:** `string`
**Required:** No (optional)
**Format:** Recommended `cmrt_{16-char-hex}` (e.g. `cmrt_a1b2c3d4e5f67890`)
**Constraints:** Must match `/^cmrt_[a-f0-9]{16}$/`. Other formats accepted by validators but flagged with an **info** signal (publishers using non-standard tokens lose registry-claim interop with citemaps.org but the field is otherwise harmless).

### Semantics

- The token is a publisher-chosen identifier embedded in the citemap to assert "this deployment is authoritative for this entry on the citemaps.org registry."
- The token is **not a secret**. It is published in a public document. It does not authenticate API calls or grant access to anything beyond the claim flow.
- The token's security model is **proof of authorship**: a registry claim succeeds only if the claimant can demonstrate that the token they submit is also present in the citemap at the published URL. Since publishing the token requires control of the citemap's deployment, this is structurally equivalent to proving control of the deployment.
- Tokens are **single-purpose**: one token per citemap deployment. Rotating the token (publishing a new value at the same URL) invalidates prior claims and enables transferring ownership (Phase 4.5 of the citemaps.org registry roadmap).

### Example

```json
{
  "@type": "Citemap",
  "version": "3.2.1",
  "citationContract": {
    "preferredName": "Western Filament",
    "registryToken": "cmrt_a1b2c3d4e5f67890"
  },
  "brand": {
    "name": "Western Filament"
  }
}
```

## 2. Producer Behavior

Producers (Studio editors, schema generators, manual authors) MAY include `citationContract.registryToken` when emitting a citemap. Producers that don't include it (legacy producers, hand-written citemaps) remain fully valid.

When a producer includes the field, it SHOULD:

- Generate the token using a cryptographically-random source (e.g. `crypto.randomBytes(8).toString("hex")` → prefix with `cmrt_`).
- Persist the token alongside the producer's own copy of the citemap source so subsequent regenerations emit the same token (rotating the token unintentionally invalidates registry claims).
- Surface the token in producer UI so the publisher knows what to paste into the claim form on citemaps.org.

The Studio reference implementation (`citemaps.ai`) generates tokens lazily on the first citemap regeneration after this spec ships, persists them on the `Property` entity, and surfaces them via the Schema → Citemap Studio tab's "Claim on citemaps.org" CTA.

---

## 3. Consumer Behavior

Consumers (validators, AI engines, registry workers) SHOULD:

- Treat the field as opaque metadata. Do not interpret the token's value beyond format validation.
- Make no inference about a citemap's authenticity based on the presence or absence of the token. An unclaimed citemap is not necessarily unverified or untrusted — many publishers will never claim their entry on citemaps.org.
- Specifically, AI engines using citemap data for citation SHOULD NOT weight `citationContract.registryToken`-having citemaps differently from those without. The token is a registry-interop field, not a trust signal in the AI-grounding pipeline.

The citemaps.org registry consumer is the one exception: it extracts the token to compare against claim submissions, surfaces a "verified" badge on the registry entry once a successful claim has occurred, and exposes the claimant's display name (publisher-set, not the email) on the entry's public detail page.

## 4. Security Considerations

### Token leakage

Because the token is published in a public document, it is by definition observable to anyone who fetches the citemap. The threat model is not "keep the token secret"; it is "prove that whoever submits the token also controls the citemap at the published URL."

The citemaps.org claim flow validates this by re-fetching the citemap from the publisher's URL at claim time and confirming the submitted token matches the live citemap. An attacker who observes a token but cannot publish a citemap at the URL cannot complete a claim (the re-fetch fails to match if the attacker controls a different deployment).

### Token rotation

Rotating the token by publishing a new value at the same URL invalidates prior claims. The intended use case is transfer of publisher control (e.g. company changes hands, CMS migrates, deployment moves). Rotation is not a periodic-security measure — there is nothing to rotate against.

### Two-factor proof

The citemaps.org claim flow adds a second factor on top of token-matching: a magic-link email to the address the claimant submitted, requiring control of that email to confirm the claim. The token proves "I control this citemap deployment"; the magic-link proves "I am at this email address." Both are required for a successful claim.

---

## 5. Validator Updates

The `@citemap/validator` SHOULD:

- Accept `citationContract.registryToken` as a valid optional field on `citationContract`.
- Emit an **info** signal when the token does not match the recommended `cmrt_{16-char-hex}` format (publishers retain the right to use other token formats; they just lose registry-claim interop).
- Make no other inference. The field is opaque metadata to the validator.
