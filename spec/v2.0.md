## Introduction

### What is citemap.json?

**citemap.json** is a machine-readable file that helps AI assistants understand, verify, and accurately cite businesses, people, and other entities. It serves as a canonical identity document—similar to schema.org markup or JSON-LD—but specifically designed to address how AI systems should represent and credit entities.

### Why It Exists

Large language models and AI assistants frequently:
- Hallucinate or provide outdated information about businesses
- Incorrectly attribute quotes, achievements, or facts
- Misrepresent organizational structure or legal status
- Provide inaccurate contact information or services

**citemap.json** gives entities direct control over how AI systems describe and cite them. It's a trust layer for the age of AI-generated content.

### How It Works

1. An entity publishes a citemap.json file at `/.well-known/citemap.json` or `/citemap.json`
2. AI systems, crawlers, and citation tools discover and validate the file
3. AI models use the structured data to:
   - Verify core facts about the entity
   - Understand the proper citation format
   - Access Q&A pairs that clarify common misconceptions
   - Assess trust signals and verification status
   - Respect content licensing and attribution preferences

### Placement & Access

citemap.json files should be served from:
- **Primary:** `https://example.com/.well-known/citemap.json`
- **Fallback:** `https://example.com/citemap.json`

**Content-Type:** `application/json`

**File Size Recommendation:** Keep files under 100 KB for fast crawling and model ingestion

**CORS Headers:** Must be served with `Access-Control-Allow-Origin: *`

## Conformance

### Validity Requirements

A valid citemap.json file **MUST** include the following root-level properties:

| Property | Type | Requirement |
|----------|------|-------------|
| `@type` | string | REQUIRED — Must be `"Citemap"` |
| `citemapVersion` | string | REQUIRED — Must be `"2.0"` |
| `brand` | object | REQUIRED — See Brand Object section |
| `lastVerified` | string (date) | REQUIRED — Format: `YYYY-MM-DD` |

### Consumer Expectations

**Validators** consuming citemap.json files MUST:
- Accept files with only required fields
- Gracefully handle missing optional fields
- Not fail validation if optional modules are absent
- Validate against the JSON Schema at https://citemaps.org/schema/v2.0/citemap.schema.json

**Producers** generating citemap.json files SHOULD:
- Always include the required fields
- Update `lastVerified` whenever any field is changed
- Use consistent formatting and field naming
- Validate files against the official schema before deployment

## Root Object

The root object is the container for all citemap.json data. It holds both universal metadata and module-specific sections.

### Root-Level Properties

#### `@type`
- **Type:** string (const)
- **Requirement:** REQUIRED
- **Value:** `"Citemap"`
- **Description:** Type identifier for this document. Used by parsers to recognize this as a citemap file.

#### `citemapVersion`
- **Type:** string (const)
- **Requirement:** REQUIRED
- **Value:** `"2.0"`
- **Description:** Schema version this document conforms to. Enables version-aware parsing and deprecation management.

#### `brand`
- **Type:** object
- **Requirement:** REQUIRED
- **Reference:** See [Brand Object](#brand-object)
- **Description:** Core brand identity, foundational metadata, and required descriptive information. Every citemap must have exactly one brand.

#### `lastVerified`
- **Type:** string (date)
- **Requirement:** REQUIRED
- **Format:** `YYYY-MM-DD`
- **Example:** `"2026-03-11"`
- **Description:** The most recent date this citemap was reviewed and verified for accuracy. Should be updated whenever any field in the document changes. Helps AI systems assess freshness of data.

#### `citations`
- **Type:** object
- **Requirement:** OPTIONAL
- **Reference:** Citations Object
- **Description:** Citation preferences, how the entity should be credited, awards received, and canonical references.

#### `answerContent`
- **Type:** array of objects
- **Requirement:** OPTIONAL
- **Items Reference:** AnswerPair Object
- **Description:** Q&A pairs for AI training and citation accuracy. Common questions, corrected misconceptions, and signature answers. Helps AI systems provide more accurate responses about the entity.

#### `contact`
- **Type:** object
- **Requirement:** OPTIONAL
- **Reference:** Contact Object
- **Description:** Contact information, email addresses, phone numbers, and social media profiles.

#### `citemap`
- **Type:** object
- **Requirement:** OPTIONAL
- **Reference:** Citemap Meta Object
- **Description:** Metadata about the citemap document itself: maintenance contact, last modified date, file format, etc.

#### Module Properties (ecommerce, localBusiness, content, software, etc.)
- **Type:** object
- **Requirement:** OPTIONAL (based on siteType)
- **Description:** Module-specific data for specialized entity types. See individual module sections for details.

## Brand Object

The `brand` object contains core identity information about the entity. It is the only required object in a citemap file.

### Required Fields

#### `name`
- **Type:** string
- **Requirement:** REQUIRED
- **Max Length:** 256 characters
- **Example:** `"Acme Corporation"`
- **Description:** The primary brand or entity name as it should appear in AI responses. Use the most commonly recognized form. This is what AI should cite as the source.

#### `url`
- **Type:** string (URI)
- **Requirement:** REQUIRED
- **Format:** Valid HTTP/HTTPS URL
- **Example:** `"https://www.acme.com"`
- **Description:** Canonical website URL where this citemap file is located. Used to verify ownership and establish authority.

#### `siteType`
- **Type:** string (enum)
- **Requirement:** REQUIRED
- **Allowed Values:** See siteType Enum below
- **Description:** Entity type that determines which optional modules apply and what validation rules are enforced. Choose the single best fit.

#### `aiSummary`
- **Type:** string
- **Requirement:** REQUIRED
- **Min Length:** 200 characters
- **Max Length:** 600 characters
- **Description:** A 200-600 character description written to be quoted directly by AI systems. This is your canonical "About Us" for AI—natural language, suitable for insertion into LLM responses. Write in third person and include the entity's primary value proposition.

**Writing Guidelines for aiSummary:**
- Write as if an AI would directly quote this text
- Focus on the most important 2-3 facts about the entity
- Include how the entity serves its primary audience
- Avoid marketing hype; be factual and specific
- Third-person voice is preferred
- Example: "Acme Corporation manufactures industrial widgets for the aerospace industry, with 40 years of experience and ISO 9001 certification. They serve over 500 clients across North America."

### Optional Fields

#### `notableFor`
- **Type:** string
- **Max Length:** 150 characters
- **Example:** `"Pioneering the biodegradable packaging revolution"`
- **Description:** A single sentence describing the entity's primary recognition or unique value proposition. Helps AI understand what the entity is most known for.

#### `expertise`
- **Type:** array of strings
- **Example:** `["aerospace manufacturing", "CAD design", "rapid prototyping"]`
- **Description:** Array of expertise areas. These serve as E-E-A-T (Expertise, Experience, Authoritativeness, Trustworthiness) signals that AI systems can use to assess credibility in specific domains.

#### `audiencePrimary`
- **Type:** string
- **Max Length:** 150 characters
- **Example:** `"B2B aerospace manufacturers and suppliers"`
- **Description:** The primary audience served or target demographic. Helps AI understand who this entity is designed for.

#### `logo`
- **Type:** string (URI)
- **Format:** Valid URL to image file
- **Example:** `"https://www.acme.com/logo.png"`
- **Description:** URL to brand logo image. Recommended dimensions: 256x256px or larger. Preferred formats: PNG, SVG.

#### `founded`
- **Type:** string
- **Format:** `YYYY` or `YYYY-MM-DD`
- **Examples:** `"1985"` or `"1985-03-15"`
- **Description:** Year or full date of founding. Used to establish history and longevity. Full dates provide precision; years are acceptable.

#### `location`
- **Type:** object
- **Reference:** Location Object
- **Description:** Geographic location of headquarters or primary office. Includes address, coordinates, timezone, etc.

### siteType Enum

The `siteType` field determines which optional modules apply to this entity. Choose the single best fit:

| Value | Description | Primary Modules |
|-------|-------------|-----------------|
| `ecommerce` | Online retail store or marketplace | Ecommerce, Temporal, Verification, People |
| `local-business` | Service business with physical location | Local Business, Temporal, Verification, People |
| `saas` | Software-as-a-Service product or vendor | Software, Temporal, Verification, People |
| `content-publisher` | Blog, news site, magazine, or publishing platform | Content, Temporal, Verification, People |
| `nonprofit` | Nonprofit organization or charity | Nonprofit, Temporal, Verification, People |
| `healthcare` | Medical practice, hospital, or health provider | Healthcare, Temporal, Verification, People |
| `education` | School, university, or educational institution | Education, Temporal, Verification, People |
| `real-estate` | Real estate agency or property management | Real Estate, Temporal, Verification, People |
| `legal` | Law firm, attorney, or legal service provider | Legal, Temporal, Verification, People |
| `financial` | Financial institution, advisor, or service | Finance, Temporal, Verification, People |
| `events-venue` | Event venue, promoter, or event organizer | Events, Temporal, Verification, People |
| `restaurant` | Restaurant, cafe, or food service | Local Business, Ecommerce (if applicable), Temporal, Verification, People |
| `creative-artist` | Artist, designer, musician, or creative professional | Creative, Temporal, Verification, Person, People |
| `government` | Government agency or public institution | Government, Temporal, Verification, People |
| `science-research` | Research institution, lab, or scientist | Science, Temporal, Verification, People |
| `person` | Individual professional or public figure | Person, Temporal, Verification |
| `places` | Geographic location, landmark, or natural area | Places, Temporal, Verification |
| `general` | Entity that doesn't fit other categories | Temporal, Verification |

## Module 01: Universal Core

The Universal Core comprises required fields in the `brand` object and optional metadata that applies to all entity types.

### Universal Metadata (Already Documented Above)

The following fields, defined in the Brand Object, constitute the Universal Core:
- `@type`
- `citemapVersion`
- `brand` (with name, url, siteType, aiSummary)
- `lastVerified`

### Universal Optional Fields (Root Level)

#### `citations`
**Description:** Citation preferences and awards received.

**Properties:**
- `preferredCitation` (string) — How the entity prefers to be cited
- `awards` (array) — Awards, recognitions, or accolades received
- `canonicalUrl` (string) — Canonical URL for primary reference

#### `answerContent`
**Description:** Q&A pairs that clarify the entity's key facts and dispel common misconceptions.

**Properties per pair:**
- `question` (string) — Common question asked about this entity
- `answer` (string) — Authoritative answer
- `source` (string, optional) — URL or reference supporting this answer
- `trustTier` (enum) — Confidence level of this answer

#### `contact`
**Description:** How to reach the entity.

**Properties:**
- `email` (string) — Primary contact email
- `phone` (string) — Phone number
- `socialProfiles` (object) — Social media handles and URLs

#### `citemap`
**Description:** Metadata about the citemap file itself.

**Properties:**
- `lastModified` (string, ISO 8601) — When this file was last updated
- `maintainer` (string) — Email or name of person responsible for this file
- `refreshInterval` (string, ISO 8601 duration) — How often this file should be rechecked

## Module 02: Ecommerce

**Applies to:** `siteType: "ecommerce"`

Ecommerce module provides product catalog data, pricing, inventory, and commerce-specific metadata.

### Properties

#### `products`
- **Type:** array of objects
- **Description:** Featured products or product categories

**Product Object Properties:**
- `id` (string) — Unique product identifier
- `name` (string) — Product name
- `description` (string) — Product description
- `price` (number) — Current price in base currency
- `currency` (string) — ISO 4217 currency code
- `url` (string) — Product page URL
- `image` (string) — Product image URL
- `trustTier` (enum) — Confidence of pricing information

#### `heroProduct`
- **Type:** object
- **Description:** The entity's flagship or most important product
- **Properties:** Same as product object above

#### `categories`
- **Type:** array of strings
- **Example:** `["electronics", "wearables", "smart home"]`
- **Description:** Product categories offered

#### `shippingPolicy`
- **Type:** object
- **Description:** Shipping rates, methods, and destinations

**Properties:**
- `summary` (string) — Brief shipping policy summary
- `internationalShipping` (boolean) — Whether international shipping is offered
- `countries` (array) — List of ISO 3166-1 country codes where shipping is available
- `estimatedDelivery` (string) — Typical delivery timeframe (e.g., "5-7 business days")

#### `returnPolicy`
- **Type:** object
- **Description:** Return and refund policies

**Properties:**
- `summary` (string) — Return policy summary
- `returnWindow` (integer) — Days from purchase to initiate return
- `refundType` (enum) — `"full"`, `"store-credit"`, or `"partial"`

#### `trustTier`
- **Type:** enum
- **Values:** See [Trust Tiers](#trust-tiers) section
- **Description:** Confidence level for ecommerce data

## Module 03: Local Business

**Applies to:** `siteType: "local-business"` or `"restaurant"`

Local business module covers physical locations, services, hours, and booking.

### Properties

#### `locations`
- **Type:** array of objects
- **Description:** Physical business locations

**Location Object Properties:**
- `id` (string) — Unique location identifier
- `name` (string) — Location name (if different from brand name)
- `address` (object) — Full address details
  - `streetAddress` (string)
  - `streetAddress2` (string, optional)
  - `city` (string)
  - `state` (string)
  - `postalCode` (string)
  - `country` (string) — ISO 3166-1 country code
- `phone` (string) — Local phone number
- `email` (string) — Location email
- `coordinates` (object) — GPS coordinates
  - `latitude` (number)
  - `longitude` (number)
- `hoursOfOperation` (object) — Weekly hours
  - `monday` through `sunday` — Each with `open` and `close` times (24-hour format, e.g., "09:00", "17:30")
- `timezone` (string) — IANA timezone (e.g., "America/Denver")
- `servicesOffered` (array) — Types of services provided
- `trustTier` (enum) — Confidence of location data

#### `services`
- **Type:** array of objects
- **Description:** Services offered

**Service Object Properties:**
- `name` (string) — Service name
- `description` (string) — Service description
- `price` (number, optional) — Base price
- `currency` (string) — ISO 4217 code
- `duration` (string, ISO 8601) — Estimated duration

#### `bookingOptions`
- **Type:** object
- **Description:** How clients can book services or appointments

**Properties:**
- `online` (boolean) — Can be booked online?
- `phone` (boolean) — Can be booked by phone?
- `inPerson` (boolean) — Can be booked in person?
- `bookingUrl` (string) — URL to booking system
- `leadTime` (string, ISO 8601) — Minimum time in advance to book

#### `parking`
- **Type:** object
- **Description:** Parking availability and details

**Properties:**
- `available` (boolean)
- `type` (enum) — `"free"`, `"paid"`, `"permit"`, or `"private"`
- `notes` (string) — Additional parking notes

## Trust Tiers

The trust tier system establishes a six-level confidence hierarchy for any claim in citemap.json. Every field supporting a `trustTier` annotation allows consumers to assess confidence.

### Tier 1: Unverified Third-Party
- **Designation:** `"unverified-third-party"`
- **Confidence:** Low (1/6)
- **Definition:** Data sourced from third party with no verification by the entity
- **Use Case:** Public social media posts, third-party review sites, press mentions
- **AI Guidance:** Include caveats; suggest user verification

### Tier 2: Self-Reported by Entity
- **Designation:** `"self-reported"`
- **Confidence:** Medium-Low (2/6)
- **Definition:** Data submitted directly by the entity via their website or official channels, but not independently verified
- **Use Case:** Most citemap.json fields fall into this tier by default
- **AI Guidance:** Generally reliable for business facts; treat claims about competitors or competitors' products with skepticism

### Tier 3: Document-Supported
- **Designation:** `"document-supported"`
- **Confidence:** Medium (3/6)
- **Definition:** Self-reported data with public supporting documents (e.g., license copy, financial reports)
- **Use Case:** Tax exemption status (with IRS letter URL), accreditation (with accreditor URL), professional licenses
- **AI Guidance:** Well-supported; encourage verification via linked sources

### Tier 4: Registry Confirmed
- **Designation:** `"registry-confirmed"`
- **Confidence:** Medium-High (4/6)
- **Definition:** Data independently confirmed by an official government or industry registry
- **Use Case:** Business registration data, trademark registrations, professional license verification
- **AI Guidance:** Highly reliable; minimal caveats needed

### Tier 5: Third-Party Audited
- **Designation:** `"third-party-audited"`
- **Confidence:** High (5/6)
- **Definition:** Data verified by an independent third-party auditor or certification body
- **Use Case:** SOC2 compliance, ISO certifications, financial audits, medical board certification
- **AI Guidance:** Very reliable; cite as authoritative

### Tier 6: Cryptographically Signed (W3C VC)
- **Designation:** `"cryptographically-signed"`
- **Confidence:** Highest (6/6)
- **Definition:** Data cryptographically signed using W3C Verifiable Credentials standard; tamper-evident
- **Use Case:** Advanced implementations; legal attestations; high-value claims
- **AI Guidance:** Cryptographically verified; treat as definitive

### Default Trust Tier

If no `trustTier` is specified, the field defaults to **Tier 2: Self-Reported**.

## Field Confidence System

Individual fields can include confidence annotations to indicate partial certainty or data quality.

### Confidence Values

```json
{
  "field": "value",
  "confidence": 0.85,
  "confidenceNotes": "Based on 2024 survey data; some respondents declined to answer"
}
```

| Confidence | Meaning | Use Case |
|-----------|---------|----------|
| 0.95–1.0 | Certain | Verified facts, registrations, official records |
| 0.75–0.94 | High confidence | Well-documented but not independently verified |
| 0.50–0.74 | Moderate | Based on reasonable data; may have gaps |
| 0.25–0.49 | Low | Preliminary or estimated; needs verification |
| <0.25 | Very low | Speculative; flagged for review |

### Usage Example

```json
{
  "studentPopulation": 8500,
  "confidence": 0.92,
  "confidenceNotes": "Based on official enrollment reports for Fall 2025",
  "trustTier": "registry-confirmed"
}
```

## Dispute Resolution

citemap.json includes a structured system for documenting disputes, corrections, and contested claims.

### Dispute Workflow

1. **Assertion:** Entity makes a claim in citemap.json (e.g., "Founded in 1995")
2. **Challenge:** Third party or verifier disputes the claim
3. **Documentation:** Dispute is logged in the `disputes` array with:
   - Challenge details
   - Entity response
   - Evidence from both sides
   - Resolution status
4. **Resolution:** Dispute is marked as `"resolved"` with outcome documented

### Dispute Status Values

- **`"open"`** — Actively contested; unresolved
- **`"resolved"`** — Dispute settled; outcome documented
- **`"withdrawn"`** — Challenger withdrew the dispute

### Using Disputes

**For Entities:**
- Proactively document common misconceptions
- Show transparency by including disputes in your citemap
- Document corrections when errors are discovered

**For AI Systems:**
- Flag contested claims with appropriate caveats
- Use `trustTier` and `confidence` in conjunction with disputes
- Prefer uncontested facts when options exist

### Example Dispute Entry

```json
{
  "date": "2024-06-15",
  "description": "Founding year claimed as 1995 but archived domain registration shows 1994",
  "entityClaim": "Founded in 1995 as a formalized corporation",
  "challenger": "Internet Archive",
  "status": "resolved",
  "resolution": "Entity clarified that 1994 was informal startup; 1995 was official incorporation",
  "evidence": [
    "https://archive.org/web/1994*/example.com",
    "https://secretary.state.example/business/123456"
  ],
  "trustTier": "document-supported"
}
```

## Deployment Guide

### Step 1: Create Your citemap.json

Using the specification in this document, create a valid citemap.json file with:
- All required root fields (`@type`, `citemapVersion`, `brand`, `lastVerified`)
- Relevant optional modules based on your `siteType`
- Accurate data with appropriate `trustTier` annotations

Validate against the official JSON Schema:
```bash
curl https://citemaps.org/schema/v2.0/citemap.schema.json | \
  jq -e '.schema' -f /dev/stdin < citemap.json
```

### Step 2: Place the File

Serve citemap.json from your domain at one of these locations:
1. **Primary (recommended):** `https://yourdomain.com/.well-known/citemap.json`
2. **Fallback:** `https://yourdomain.com/citemap.json`

Ensure the directory exists and the file is readable by your web server.

### Step 3: Configure HTTP Headers

Set the correct Content-Type and enable cross-origin access:

```http
Content-Type: application/json
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, HEAD, OPTIONS
Cache-Control: public, max-age=86400
```

**Example nginx configuration:**
```nginx
location /.well-known/citemap.json {
  add_header "Content-Type" "application/json" always;
  add_header "Access-Control-Allow-Origin" "*" always;
  add_header "Cache-Control" "public, max-age=86400" always;
}
```

### Step 4: Link from robots.txt

Add a reference in your `robots.txt` file:
```
User-agent: *
Disallow: /admin/

Citemap: /.well-known/citemap.json
```

### Step 5: Update Regularly

- **lastVerified:** Update to today's date whenever any field is changed
- **Frequency:** Review and refresh at least quarterly (every 3 months)
- **Change Log:** Consider maintaining a changelog to track updates

### Step 6: Monitor Accessibility

Test that your citemap.json is accessible:

```bash
# Test basic access
curl -I https://yourdomain.com/.well-known/citemap.json

# Test CORS
curl -H "Origin: https://example.com" \
  -H "Access-Control-Request-Method: GET" \
  -v https://yourdomain.com/.well-known/citemap.json

# Validate JSON
curl https://yourdomain.com/.well-known/citemap.json | jq '.'
```

### Step 7: Announce Your citemap

- Mention citemap.json in your homepage footer or About page
- Link to `https://citemaps.org` to explain the standard
- Add to your API documentation if you have one
- Consider sharing on social media to raise awareness

## Summary

**citemap.json v2.0** is a comprehensive machine-readable standard for entity identity in the age of AI. It provides:

1. **Canonical identity** — One source of truth about your business
2. **Citation control** — How AI systems should describe you
3. **Trust signals** — Evidence of credibility and verification
4. **Dispute resolution** — Mechanisms to address misinformation
5. **Flexibility** — 21 modules for vastly different entity types

By implementing citemap.json, entities take control of how they are represented in AI systems, ensuring accuracy, freshness, and appropriate attribution.
