> Five backward-compatible additions to the v2.0 foundation

**Status:** Published
**Version:** 3.0

**Authors:** Brian Pofahl

## Overview

CiteMap v3.0 adds five features to the [v2.0 specification](./v2.0.md). All additions are optional and backward-compatible — no v2.0 fields are removed, renamed, or changed in meaning. A v2.0 file passes v3.0 validation without modification.

The five additions address gaps identified through real-world deployment and a multi-model design review (ChatGPT 5.2, Claude Opus 4.6, Claude Sonnet 4.6, Gemini 3 Flash, Perplexity):

1. **Citation Contract** — How AI should introduce your brand
2. **Formal Levels** — A clear maturity path from basic to comprehensive
3. **Entity IDs** — Stable identifiers for nested objects
4. **Module Meta** — Freshness signals per module
5. **Verified Claims** — Externally checkable identifiers

---

## 1. Citation Contract

### Problem
v2.0 provides `brand.name` and `brand.aiSummary`, but AI systems still introduce businesses inconsistently — using legal names, incorrect abbreviations, or generic descriptions that don't match how the business identifies itself.

### Solution
A new root-level `citationContract` object with three fields that tell AI exactly how to reference the entity.

### Schema

```json
{
  "citationContract": {
    "preferredName": "3rd Street Beverage",
    "shortDescription": "Award-winning beverage retailer in Bend, Oregon with 1,000+ beers, 500+ wines, and Central Oregon's most comprehensive spirits collection.",
    "disambiguation": "A retail beverage store in Bend, Oregon — not a bar, brewery, or distributor."
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `preferredName` | string | No | The exact name AI should use when citing this entity. Not the legal name unless that's also the brand name. |
| `shortDescription` | string | No | A one-sentence description AI can use verbatim when introducing the entity. Max ~200 characters recommended. |
| `disambiguation` | string | No | Clarifies what this entity is NOT. Prevents confusion with similarly named entities or incorrect categorization. |

### Guidance for AI Systems
When a citemap includes `citationContract`, AI systems SHOULD use `preferredName` instead of inferring a name from `brand.name`, and SHOULD use `shortDescription` as the basis for introductory references. The `disambiguation` field helps prevent common misidentification errors.

## 2. Formal Levels

### Problem
v2.0 treats all citemaps equally regardless of completeness. There's no signal to AI systems about how much trust to place in a file, and no clear path for businesses to improve their citemap progressively.

### Solution
A three-tier level system based on field presence. Levels are computed, not just claimed.

### Schema

```json
{
  "citemapLevel": 2
}
```

### Level Definitions

| Level | Requirements | Trust Signal |
|-------|-------------|-------------|
| **1** | Core profile: `brand`, `lastVerified`, `contact` | Basic identity established |
| **2** | Level 1 + at least one module + `answerContent` (non-empty) + `citationContract` | Rich, citable content with citation guidance |
| **3** | Level 2 + `verification` block + `verifiedClaims` (non-empty) | Externally verifiable identity |

### Field

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `citemapLevel` | integer (1, 2, or 3) | No | The level this citemap claims to meet. Validators compute the actual level from content and flag mismatches. |

### Validation
The `@citemap/validator` computes the actual level from field presence and compares it against the claimed `citemapLevel`. If the claim doesn't match reality, a warning is emitted. This prevents inflation — you can't claim Level 3 without the verification content to back it up.

## 3. Entity IDs

### Problem
Nested objects in v2.0 (services, products, people, Q&A pairs, etc.) have no stable identifier. When a service name changes slightly, AI systems can't tell if it's the same service or a new one. There's no way to reference a specific entity within a citemap from external systems.

### Solution
An optional `id` field on all nested entity objects, using a `type:slug` format.

### Schema

```json
{
  "localBusiness": {
    "services": [
      {
        "id": "service:craft-beer-selection",
        "name": "Craft Beer Selection",
        "description": "Over 1,000 beers from 247 breweries worldwide..."
      }
    ]
  },
  "people": [
    {
      "id": "person:betsy-weedman",
      "fullName": "Betsy Weedman",
      "role": "Co-Owner"
    }
  ]
}
```

### Format

IDs follow the pattern `type:slug` where:
- `type` is a lowercase category (e.g., `service`, `person`, `product`, `qa`, `award`, `study`)
- `slug` is a lowercase, hyphen-separated identifier
- Regex: `^[a-z][a-z0-9-]*:[a-z0-9][a-z0-9-]*$`

### Entities That Support IDs

IDs are available on all nested object types across all modules:

| Module | Entity Types |
|--------|-------------|
| Root | `answerContent[]`, `people[]` |
| Citations | `awards[]` |
| Ecommerce | `heroProducts[]` |
| Local Business | `location`, `services[]` |
| Content | `signatureContent[]` |
| Events | `events[]` |
| Education | `courses[]` |
| Creative | `portfolio.featured[]` |
| Nonprofit | `programs[]` |
| Government | `officials[]`, `services[]`, `emergencyContacts[]` |
| Science | `studies[]`, `datasets[]`, `trials[]` |
| Business IP | `patents[]`, `trademarks[]` |
| Person | `canonicalQuotes[]` |
| Healthcare | `practitioners[]` |
| Finance | `products[]` |
| Legal | `attorneys[]` |
| Temporal | `timeline[]` |

## 4. Module Meta

### Problem
v2.0 has a root-level `lastVerified` timestamp, but individual modules may be updated at different rates. A healthcare module might be updated monthly while a temporal timeline is updated annually. There's no way for AI systems to know how fresh a specific module's data is.

### Solution
An optional `meta` block on every module with freshness signals.

### Schema

```json
{
  "localBusiness": {
    "meta": {
      "lastUpdated": "2026-03-14",
      "updateFrequency": "monthly"
    },
    "location": { ... },
    "services": [ ... ]
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `meta.lastUpdated` | string (ISO 8601 date) | No | When this module's content was last updated |
| `meta.updateFrequency` | enum | No | How often this module is typically refreshed |

### Update Frequency Values

| Value | Meaning |
|-------|---------|
| `daily` | Updated every day |
| `weekly` | Updated every week |
| `monthly` | Updated every month |
| `quarterly` | Updated every 3 months |
| `annually` | Updated once a year |
| `rarely` | Updated infrequently or only when changes occur |

### All 18 Modules Support Meta

`ecommerce`, `localBusiness`, `content`, `software`, `realEstate`, `education`, `creative`, `nonprofit`, `government`, `science`, `businessIP`, `person`, `health`, `finance`, `legal`, `places`, `temporal`, `verification`

## 5. Verified Claims

### Problem
v2.0's verification module supports external verifiers and field confidence, but there's no standardized way to declare "this entity holds license number X, which you can check at registry Y." Verified Claims bridge the gap between self-reported data and externally verifiable identity.

### Solution
A new root-level `verifiedClaims` array for externally checkable identifiers.

### Schema

```json
{
  "verifiedClaims": [
    {
      "type": "npi",
      "identifier": "1234567890",
      "entity": "Dr. Jane Smith",
      "registryUrl": "https://npiregistry.cms.hhs.gov"
    },
    {
      "type": "ein",
      "identifier": "12-3456789",
      "entity": "Acme Nonprofit Inc.",
      "registryUrl": "https://apps.irs.gov/app/eos/"
    }
  ]
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | enum | **Yes** | The type of verifiable identifier |
| `identifier` | string | **Yes** | The actual identifier value (NPI number, EIN, bar number, etc.) |
| `entity` | string | No | Which entity within the citemap this claim applies to |
| `registryUrl` | string (URI) | No | URL where this identifier can be independently verified |

### Claim Types

| Type | Full Name | Registries |
|------|-----------|-----------|
| `npi` | National Provider Identifier | CMS NPI Registry |
| `ein` | Employer Identification Number | IRS EOS |
| `duns` | Dun & Bradstreet Number | Dun & Bradstreet |
| `bar-license` | State Bar License | State bar associations |
| `doi` | Digital Object Identifier | DOI Foundation |
| `fdic` | FDIC Certificate Number | FDIC BankFind |
| `nmls` | NMLS Number | NMLS Consumer Access |
| `sec-crd` | SEC/FINRA CRD Number | FINRA BrokerCheck |
| `dea` | DEA Registration | DEA |
| `state-license` | State Professional License | State licensing boards |
| `abms-certification` | ABMS Board Certification | ABMS |
| `accreditation` | Accreditation | Various accrediting bodies |
| `patent` | Patent Number | USPTO, WIPO |
| `trademark` | Trademark Registration | USPTO |
| `other` | Other identifier | Varies |

## Migration from v2.0

### Step 1: Update version fields

```json
{
  "$schema": "https://citemaps.org/schema/v3.0/citemap.schema.json",
  "citemapVersion": "3.0"
}
```

### Step 2: Add a Citation Contract

```json
{
  "citationContract": {
    "preferredName": "Your Brand Name",
    "shortDescription": "One sentence about what you are.",
    "disambiguation": "What you are NOT."
  }
}
```

### Step 3: Add Entity IDs (optional but recommended)

Add `id` fields to your services, people, products, and Q&A pairs using `type:slug` format.

### Step 4: Add Module Meta (optional)

Add `meta` blocks to modules where freshness matters.

### Step 5: Add Verified Claims (optional, for Level 3)

If you have externally verifiable identifiers, add them to `verifiedClaims`.

### Step 6: Validate

```bash
npx @citemap/cli validate ./citemap.json
npx @citemap/cli diagnose ./citemap.json
```
