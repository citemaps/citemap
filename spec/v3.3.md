# citemap.json Specification v3.3

> The consolidation release: `@graph` extension, foundational sections, 14 ride-along field additions, and four taxonomy retirements

**Status:** Draft — scaffold
**Version:** 3.3
**Date:** 2026 (in authoring)
**Authors:** Brian Pofahl
**License:** MIT
**Schema:** https://citemaps.org/schema/v3.3/citemap.schema.json

## Overview

v3.3 is the **consolidation release**. Where v3.1 and v3.2 each added a single focused feature on top of the prior version, v3.3 packages a deliberate batch of structural additions, field-set extensions, and taxonomy cleanups that have accumulated since v3.2 shipped. The motivation: cutting one well-planned v3.3 is cheaper for the ecosystem (Yoast, Rank Math, next-seo, in-the-wild parsers, the publicly available CiteMap validator) than spreading the same set of changes across three accidental point releases.

The v3.3 cut delivers five categories of change, in order of architectural significance:

1. **JSON-LD `@graph` extension.** v3.3 makes citemap.json an explicit JSON-LD document, adding optional top-level `@graph[]` and `relationships[]` arrays for declaring multi-entity networks — franchises, parent/subsidiary chains, multi-location practices, practitioner networks, multi-vertical companies, successor/predecessor entities. The current spec describes a single entity well; v3.3 adds the vocabulary to describe how that entity relates to others. AI engines that already parse schema.org `@graph` get richer entity disambiguation; tools that don't read the extension see a v3.2-shaped document with a few additional `brand.*` fields.

2. **Foundational sections (always-on contract).** v3.3 separates "verticals" (user-selected categorical classification, optional, multi-able) from "foundational sections" (always emitted regardless of vertical selection). The v3.0 flat "modules" catalog conflated the two. v3.3 documents the contract: every citemap.json carries `brand`, `citationContract`, `temporalRecord`, `trust.perField`, `answerContent`, `brand.team`, `brand.products`, `brand.sameAs`, `policy` (new), and `verification` (promoted from optional module to foundational). Producers can no longer suppress these; consumers can rely on them being present.

3. **Constituent field additions (14 fields).** Fields that have been emitted by Studio's reference implementation ahead of the spec — `brand.legalName`, `brand.email`, `brand.taxonomy` (revisions), `brand.team[]` (cross-vertical core), `brand.teamPageUrl`, `brand.products[]`, `brand.locations[]`, `brand.subBrands[]`, `brand.productCategories[]` + `brand.serviceCategories[]`, `brand.services[]`, `brand.storefronts[]`, `brand.publications[]`, plus the per-vertical field promotions that surfaced during the 2026-Q2 multi-vertical dogfood. v3.3 documents the canonical shape, semantics, and derivation rules for each. v3.2 emitters continue to validate; v3.3 producers and consumers gain a richer agreed vocabulary.

4. **Modules → verticals + foundational reframe.** The 15 user-pickable verticals (with renames: `events` → `events-venue`, `health` → `healthcare`, `legal` → `legal-services`, `finance` → `financial-services`) replace the v3.0 module catalog as the user-facing classification layer. Legacy module keys are emitted alongside the new vertical keys for one release window so v3.0/v3.1/v3.2 consumers reading hardcoded keys continue to find them. Three v3.0 modules are dropped (`business-IP` consolidated into `brand`; `people` superseded by `brand.team`; `places` superseded by `brand.location` + per-location entries in `brand.locations[]`).

5. **Vertical taxonomy retirements.** Four entries that were structurally axis-pretending-to-be-vertical retire from the recommended `taxonomy.primaryVertical` registry: `local-business`, `manufacturer`, `membership-organization`, and `industrial-services-b2b`. Each stays as a valid derived `brand.siteType` value for back-compat (the siteType enum is unchanged), but stops being a recommended `primaryVertical` option. Producers with these as their primary vertical get a migration path to the closest real vertical with a low-confidence flag. Multi-vertical equality stays intact — no primary-vs-secondary split.

### Design Principle: additive, never breaking

Every change in v3.3 is purely additive at the structural level. A v3.2 citemap.json passes v3.3 validation with at most a one-line `version` bump. v3.3-specific fields (`@graph`, `relationships`, the constituent additions, the foundational sections) are either optional or always-emitted-by-the-emitter but optional-on-the-consumer-side. Renamed module keys are emitted alongside their legacy counterparts; taxonomy retirements affect the registry's recommended-values list but not the validator's accepted-values list.

The cost of the consolidation is that v3.3 documents are slightly fatter than v3.2 documents (the always-on foundational sections + the dual-write renamed keys), and producers eventually have to drop the legacy keys at v3.4. The benefit is that downstream tooling — schema generators, AI engine training pipelines, the citemaps.org validator, third-party parsers — gets one coordinated upgrade moment instead of three.

### Design Principle: JSON-LD native, not graph-format-native

v3.3 adopts schema.org's `@graph` mechanism for multi-entity documents rather than introducing a generic graph format (JSON Graph Format, GraphSON, etc.). The rationale is adoption-shaped: JSON-LD `@graph` is what every AI engine training pipeline already parses, what every SEO tool that emits structured data already understands, and what every schema.org-compliant Knowledge Graph indexes from. Picking a graph format the engines don't natively read defeats citemap.json's purpose. The architectural decision and the counterarguments are documented at length in [§1.1](#11-why-json-ld-graph-and-not-json-graph-format).

### Design Principle: foundational always-on as a differentiation moat

The decision to make 10 sections always-on (rather than user-toggleable per the v3.0 module model) is partly architectural cleanup and partly strategic. AI engines reading a citemap.json get a guaranteed minimum of structured signal: identity (`brand`), provenance (`citationContract`), temporal record (`temporalRecord`), trust (`trust.perField`), authored answers (`answerContent`), team (`brand.team`), products (`brand.products`), social presence (`brand.sameAs`), publisher trust (`policy`), and verification claims (`verification`). Competitors emitting bare schema.org Organization look thin next to a citemap that carries 10 foundational sections per regen. The "always-on foundational" framing is concrete and pitchable in marketing copy and as differentiated structural signal AI engines weight.

## 1. JSON-LD `@graph` extension

The centerpiece structural addition of v3.3. v3.3 makes citemap.json an explicit JSON-LD document, with optional top-level `@graph[]` and `relationships[]` arrays that let a citemap describe a network of entities and the relationships between them — not just a single entity. The current spec describes a single entity well; this section adds the vocabulary for multi-entity networks.

The keywords MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD, SHOULD NOT, RECOMMENDED, MAY, and OPTIONAL in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

### 1.1 Why JSON-LD `@graph` and not JSON Graph Format

Two well-defined options exist for multi-entity JSON documents:

- **[JSON Graph Format](https://jsongraphformat.info/)** — a generic graph spec with clean semantics for nodes and edges. Well-defined, well-documented, and parsed by zero AI engines natively.
- **JSON-LD `@graph`** — the mechanism schema.org uses for multi-entity documents (it's how Yoast emits a single JSON-LD block containing WebSite, Organization, WebPage, and BreadcrumbList entities at once). Every AI engine training pipeline parses it. Every SEO tool that emits structured data understands it. The World Wide Web Consortium standardized it. AI engines have been seeing it for a decade.

**v3.3 adopts JSON-LD `@graph`.** The architectural decision is adoption-shaped: picking a format AI engines don't natively read defeats citemap.json's purpose. The verbosity tax of JSON-LD (`@context`, `@id`, `@type` everywhere) is the price of being parsed natively by the systems that matter. Producers that want a lighter wire format are free to mint their own; citemap.json is in the business of being adopted, not of being elegant.

This means v3.3 documents are valid JSON-LD documents end to end. They were already JSON-LD-shaped under v3.2; v3.3 makes the JSON-LD nature explicit by adopting `@context` and `@graph` as first-class top-level fields.

### 1.2 Top-level v3.3 document shape

```jsonc
{
  "@context": "https://citemaps.org/v3.3",
  "version": "3.3",
  "self": { /* validator + meta, unchanged from v3.2 */ },
  "brand": {
    "@type": "Organization",       // or LocalBusiness / specific subtype
    "@id": "https://example.com/#org",
    "name": "Acme HVAC",
    "legalName": "Acme HVAC, LLC",
    "siteType": "local-business",
    "taxonomy": { /* unchanged from v3.2 + revisions per §6 */ },
    /* ...existing v3.2 brand fields + constituent additions per §4... */
    "products": [ /* see §4.7 */ ],
    "services": [ /* see §4.11 */ ],
    "team": [ /* see §4.5 */ ],
    "sameAs": [ /* see §4.3 */ ],
    "publicEmail": "hello@example.com",
    "expertise": [ /* authored-content registry */ ]
  },
  "modules": { /* localBusiness / ecommerce / etc., unchanged from v3.2 */ },
  /* ─── OPTIONAL v3.3 graph extension ─── */
  "@graph": [
    {
      "@type": "Organization",
      "@id": "https://example.com/#parent-org",
      "name": "Acme Holdings LLC"
    },
    {
      "@type": "Person",
      "@id": "https://example.com/#owner",
      "name": "Jane Doe",
      "jobTitle": "Owner"
    }
  ],
  "relationships": [
    {
      "from": "https://example.com/#org",
      "to":   "https://example.com/#parent-org",
      "type": "subOrganizationOf",
      "verified": false
    },
    {
      "from": "https://example.com/#owner",
      "to":   "https://example.com/#org",
      "type": "founderOf",
      "verified": true,
      "observedAt": "2026-05-31"
    }
  ]
}
```

**Backward compatibility.** Both `@graph` and `relationships` are OPTIONAL. A v3.3 document with neither is functionally identical to a v3.2 document plus the constituent field additions from §4. Tools that do not recognize the graph extension MUST be able to read v3.3 documents and treat them as v3.2 + additive fields; they will not see the entity network but will not break.

The `@context` value and the `version` bump to `"3.3"` are the only structural changes required of any existing v3.2 emitter to claim v3.3 conformance.

### 1.3 `@graph[]` array semantics

The `@graph[]` array contains entities that are part of the primary entity's published network but are not the primary subject of this citemap. Each node MUST be a valid schema.org-shaped object with at minimum `@type` and `@id`.

- A node SHOULD describe an entity that the primary entity has a real relationship with — a parent corporation, a subsidiary, a location, a practitioner, a marketplace storefront, a predecessor entity. Nodes that have no connection to the primary entity in `relationships[]` are orphans (see below) and MUST be rejected by validators.
- Nodes MAY be sparse — for many use cases the `@id` + `name` + `@type` is enough. Full brand-shaped descriptions are appropriate for entities the primary entity wants to canonically represent (a wholly-owned subsidiary's brand profile); minimal descriptors are appropriate for entities the primary entity merely references (a parent corporation whose own citemap exists at a different URL).
- Cross-document `@id` references — entities whose `@id` is an absolute URL to a node defined in another citemap.json — MUST be valid. The validator (§9.4) resolves verification status by fetching the counter-party citemap.

**Orphan nodes are an error.** Every node in `@graph[]` MUST be reachable from the primary entity via at least one edge in `relationships[]`, directly or transitively. A `Person` entity in `@graph` with no edges to anything is malformed — the entity has no connection to the primary subject, so it does not belong in this citemap.

### 1.4 `relationships[]` array semantics

The `relationships[]` array contains edges between the primary entity, nodes in `@graph[]`, and (optionally) entities defined in other citemaps via cross-document `@id` references. Each edge is an object with the following fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `from` | string (`@id`) | **Yes** | Source entity. MUST point to the primary entity's `@id`, a node in this document's `@graph[]`, or an absolute URL to a node in another citemap.json. |
| `to` | string (`@id`) | **Yes** | Target entity. Same resolution rules as `from`. |
| `type` | string | **Yes** | Edge type from the v3.3 taxonomy (§2) OR an extension URI (§2.3). |
| `verified` | boolean | No (default `false`) | Two-way confirmation status — see §1.7. |
| `observedAt` | ISO 8601 date | No | When this edge was first asserted. Producers SHOULD set this for edges representing time-sensitive relationships (employment, location, ownership). |
| `source` | string (URL) | No | Where the edge claim is evidenced (e.g. `https://example.com/about`, `https://example.com/press-release/2024-03-acquisition`). Useful for human review and validator audit trails. |
| `confidence` | enum: `high` / `medium` / `low` | No | Producer's self-rated confidence in the edge. Reserved for cases where automated extraction proposed the edge but the producer hasn't manually verified. v3.3 producers SHOULD omit this field unless they have a concrete confidence model. |
| `context` | string | No | Free-text context attached to the edge. Used by `operatesAs` self-edges (§1.8) to disambiguate the multi-vertical role the edge represents; rarely used elsewhere. |

Edges MAY carry additional implementation-defined fields. Validators SHOULD ignore unknown edge fields rather than reject documents.

### 1.5 Why `relationships` is a separate array, not inline

A natural alternative would be to express relationships inline as schema.org properties:

```jsonc
{
  "brand": {
    "@type": "Organization",
    "@id": "https://example.com/#org",
    "parentOrganization": { "@id": "https://example.com/#parent-org" }
  }
}
```

This works for the schema.org relationship types that ARE first-class properties (`parentOrganization`, `subOrganization`, `memberOf`, `worksFor`, `affiliation`, `founder`, `sameAs` — see §2.1). v3.3 mirrors those inline ([§1.6](#16-mirroring-schemaorg-native-relationships)). For other edge types it breaks down:

1. **Edge metadata has nowhere to live.** When was the edge first observed? Who asserted it? Is it verified? An inline `parentOrganization: { ... }` carries no slot for any of these.
2. **Some edge types don't exist in schema.org.** `franchiseOf`, `predecessorOf`, `operatesAs` — schema.org has no inline property for any of them. Treating them as `additionalType` or stuffing them into `description` loses the structured semantics.
3. **Direction is awkward.** Inline properties force every edge into the entity's outgoing perspective. Expressing "this entity is a member of X" inline forces you to invert when you actually want "X has member this entity" elsewhere in the document.

A separate `relationships[]` array sidesteps all three. Edges carry their own metadata cleanly, can use vocabulary outside schema.org's properties, and can express direction explicitly. Tools that only read schema.org-native properties still get the parentOrganization / subOrganization / memberOf / worksFor / affiliation / founder / sameAs inline (per §1.6); tools that want the full graph use the dedicated array.

### 1.6 Mirroring schema.org-native relationships

For the seven edge types in §2.1 that have native schema.org property mappings, v3.3 producers MUST emit the relationship in BOTH places:

- As the corresponding inline schema.org property on the entity (so generic JSON-LD parsers and schema.org-native consumers see it)
- As a row in `relationships[]` (so the graph layer is complete and the edge's metadata — `verified`, `observedAt`, `source`, `confidence` — is preserved)

Example:

```jsonc
{
  "brand": {
    "@type": "Organization",
    "@id": "https://example.com/#org",
    "name": "Acme HVAC",
    "parentOrganization": { "@id": "https://example.com/#parent-org" }
  },
  "@graph": [
    {
      "@type": "Organization",
      "@id": "https://example.com/#parent-org",
      "name": "Acme Holdings LLC"
    }
  ],
  "relationships": [
    {
      "from": "https://example.com/#org",
      "to":   "https://example.com/#parent-org",
      "type": "subOrganizationOf",
      "verified": true,
      "observedAt": "2024-08-15"
    }
  ]
}
```

This is the same redundancy pattern v3.2 already uses for `brand.taxonomy` (additive, doesn't break siteType). Slightly fatter JSON in exchange for meaningfully better cross-tool compatibility.

Validators MAY warn on mismatch — for example, a `relationships[]` row of type `subOrganizationOf` that has no corresponding `parentOrganization` inline property on the entity. Warnings MUST NOT cause validation to fail; mirroring is a SHOULD, not a MUST.

### 1.7 Cross-document edges and verification

The `verified` field on a relationship has explicit semantics:

| `verified` | Meaning |
|---|---|
| `false` (default) | One-way claim. This entity asserts the relationship; the other end has not confirmed. |
| `true` | Two-way confirmation. The other end's citemap.json (or other authoritative source) also asserts the inverse relationship. |

The citemap.org validator (and any conforming validator) is responsible for the cross-fetch check:

1. For every edge whose `to` is an absolute URL to a citemap node, the validator SHOULD fetch the referenced citemap.json.
2. The validator SHOULD inspect the counter-party document for an inverse relationship that names this document's primary entity.
3. If the inverse exists and the counter-party document is fresh (< 6 months old based on its `temporalRecord` or HTTP `Last-Modified`), the edge is verified.
4. If the inverse is absent, the document is stale, or the fetch fails, the edge stays unverified. Validators MUST NOT block emission on verification failure.

Producers SHOULD NOT set `verified: true` themselves; the field reflects validator-determined status. A producer that asserts `verified: true` without the validator's cross-check is making a claim the validator may reject on next audit. Most producers leave the field absent (defaults to `false`) and let validators flip it during their audit pass.

Unverified edges are not ignored by downstream consumers. Many legitimate relationships are one-way claims — a small business asserting "we're an authorized dealer of Toro" is one-way; Toro is unlikely to reciprocate in their citemap. AI engines and trust-scoring tools MAY weight verified edges higher than unverified, but unverified edges remain valid published signal.

### 1.8 Self-edges and the `operatesAs` multi-vertical pattern

A self-edge is a relationship whose `from` and `to` reference the same `@id`. The canonical use case is the multi-vertical entity:

```jsonc
{
  "brand": {
    "@type": "Organization",
    "@id": "https://wfilament.com/#org",
    "name": "Western Filament",
    "taxonomy": {
      "primaryVertical": "manufacturer",
      "additionalVerticals": ["industrial-services"]
    }
  },
  "relationships": [
    {
      "from": "https://wfilament.com/#org",
      "to":   "https://wfilament.com/#org",
      "type": "operatesAs",
      "context": "vertical:industrial-services"
    }
  ]
}
```

The edge declares that the primary entity operates as both a manufacturer (its `primaryVertical`) AND industrial-services (its `additionalVertical`) — and that the two are roles within one entity, not two separate companies. The `context` field disambiguates which role the self-edge represents.

Self-edges resolve a structural problem that surfaced during Studio's Phase 2.2 dogfood: multi-vertical properties were silently double-emitting per-vertical module data (a `manufacturer` block AND an `industrial-services` block with overlapping fields). The alternative representations all have downsides:

- **Multiple `@graph` Organization nodes (one per vertical)** — forces splitting one legal entity across multiple `@graph` entries. Conceptually wrong: there is one Western Filament, not two.
- **Vertical info on inline taxonomy only (current pre-v3.3 approach)** — works but loses the relationship semantics that downstream tooling needs to merge modules correctly.
- **Self-edge with `context` field** — the v3.3 approach. Slightly unusual in graph theory, but expresses the right model.

Validators SHOULD warn on self-edges of types other than `operatesAs` (the pattern is reserved for the multi-vertical case in v1). Validators MUST NOT reject self-edges outright; the `operatesAs` use case is legitimate and additional uses may emerge.

## 2. Edge taxonomy v1

The 13 relationship types v3.3 ships with, organized by whether they mirror a schema.org-native property (§2.1) or are CiteMaps extensions (§2.2). The taxonomy is deliberately small — easier to expand the vocabulary when real-world demand surfaces than to retire over-eager additions later. Producers MAY emit custom edge types using the extension URI mechanism (§2.3); validators MUST NOT reject custom types but SHOULD surface them as warnings for human review (§2.6).

Every edge type has a stable type-vocabulary URI at `https://citemaps.org/v3.3/types/{type}` that external tools can dereference to learn the type's semantics. Same pattern schema.org uses for its property URIs.

### 2.1 Schema.org-native types (mirrored inline + relationship row)

These seven edge types correspond to first-class schema.org properties. Producers MUST mirror them per §1.6: emit both the inline property on the entity AND the relationship row in `relationships[]`.

| Type | Schema.org property | Inverse | Use case |
|---|---|---|---|
| `parentOrganizationOf` | `subOrganization` | `subOrganizationOf` | "I am a parent of X" — emit on the parent's citemap pointing at the subsidiary. |
| `subOrganizationOf` | `parentOrganization` | `parentOrganizationOf` | "I am a subsidiary of X" — emit on the subsidiary's citemap pointing at the parent. |
| `memberOf` | `memberOf` | `member` (schema.org) | "I am a member of X" — associations, industry groups, professional bodies, accreditation councils. |
| `worksFor` | `worksFor` | `employee` (schema.org) | "Person → Organization" employment. Used for practitioner networks where the employment is the primary signal. |
| `affiliatedWith` | `affiliation` | (no clean inverse) | Looser association than `worksFor`. Use for advisory relationships, board memberships, fellowships, named-but-non-employee partnerships. |
| `founderOf` | `founder` | (no clean inverse) | "Person → Organization they founded" — distinct from `worksFor` because founders may no longer work for the entity. |
| `sameAs` | `sameAs` | `sameAs` (symmetric) | Same-entity identity claims. Used to point at the entity's other canonical representations: Wikipedia, Wikidata, Crunchbase, LinkedIn company page, etc. |

The inverse columns are informational; producers are responsible only for their own outgoing edges. When two entities both publish citemap.json and both correctly assert the relationship (a parent emitting `parentOrganizationOf` and the subsidiary emitting `subOrganizationOf` pointing back), the cross-document verification check (§1.7) flips both edges to `verified: true`.

### 2.2 CiteMaps-extension types (relationship row only)

These six edge types do not have first-class schema.org properties and emit only in `relationships[]`. Each carries semantics that surfaced during real-world dogfood across the entity types citemap.json aims to serve.

| Type | Use case | Notes |
|---|---|---|
| `franchiseOf` | Franchisee → franchisor | Distinct from `subOrganizationOf`: franchises are typically not legal subsidiaries, but the relationship is AEO-relevant because the franchisor brand carries citation weight the franchisee inherits. Schema.org has no first-class franchise property. |
| `locationOf` | A specific physical location of a multi-location org | Distinct from `subOrganizationOf`: locations are typically not separate legal entities. Used for dental practices, restaurant chains, retail with branches, etc. Pairs with the per-location detail emitted in `brand.locations[]` (§4.8). |
| `practitionerAt` | Person → Organization (alternate to `worksFor`) | Healthcare, legal, and professional-services context. Carries clinical/professional role semantics distinct from employment. A radiologist is a `practitionerAt` a hospital even if they're contracted rather than employed. |
| `predecessorOf` / `successorOf` | Entity continuity through rebrand, acquisition, spin-off | Schema.org has nothing for these. AI engines often need them to disambiguate "the old Acme Plumbing vs the new Acme Plumbing LLC" when reviews, citations, and content from the predecessor identity should attach to the current entity. Both directions emitted as separate edges. |
| `operatesAs` | Self-edge for multi-vertical roles within one entity | The Western Filament case (§1.8). The `context` field disambiguates which vertical role the edge represents. Self-edges of other types SHOULD warn. |
| `sellsAt` | Entity → Marketplace storefront | Entity-level marketplace presence (Amazon Brand Store, eBay storefront, Etsy shop, Walmart Marketplace, Faire, etc.). Pairs with the per-storefront detail emitted in `brand.storefronts[]` (§4.12). Bidirectional in spirit (the marketplace's seller page IS a citation back to the entity), but `verified: true` requires the marketplace's own structured data to confirm the inverse, which most marketplaces do not currently expose. v1 these stay `verified: false` self-claims. |

### 2.3 Type-vocabulary URI registry pattern

Every edge type — both schema.org-native and CiteMaps-extension — has a stable type-vocabulary URI:

```
https://citemaps.org/v3.3/types/{type}
```

External tools can dereference the URI to read the type's semantics, use cases, and any constraints. The registry is published alongside the spec at [`spec/registry/v3.3-vocabulary.md`](./registry/v3.3-vocabulary.md), with one section per type.

Producers MAY define custom edge types using their own URI namespace:

```jsonc
{
  "relationships": [
    {
      "from": "https://example.com/#org",
      "to":   "https://supplier.com/#org",
      "type": "https://example.com/vocab/preferredVendor"
    }
  ]
}
```

Custom-type URIs MUST be absolute URLs. Producers SHOULD host vocabulary documentation at the URL so external tools can resolve the semantics. Validators MUST treat custom-type edges as valid but SHOULD warn (open-world per §2.6).

When patterns emerge across multiple producers, custom types MAY graduate into the spec registry at a future version bump.

### 2.4 Types rejected for v1

The following types were considered for v1 and deliberately rejected. Each MAY return in v3.4 or later if real-world usage justifies inclusion.

| Type | Why rejected |
|---|---|
| `competitorOf` | Competitor relationships are not asserted by the entity — they're inferred by tools and observers. Belongs in CiteMaps Studio's evidence layer, not in the published citemap.json. |
| `partnerOf` | Too vague. Different industries use "partner" differently — channel partner, technology partner, strategic alliance, joint venture, vendor relationship. Punt until customer usage clarifies which subset to model. |
| `customerOf` / `supplierOf` | B2B relationship types that introduce NDA and confidentiality concerns. Belongs in private CRM data, not public citemap.json. |
| `licensedBy` / `accreditedBy` | These are actually Credential entities (already modeled in v3.2's modules via `brand.team[*].credentials` and `verification.claims[]`). They do not need new edges; they need the credential model to surface in the @graph layer when appropriate. v3.4+ may add a `credentialFrom` edge type if the credential model expands. |

### 2.5 Self-verification semantics

The `verified` field on every edge type carries the semantics defined in §1.7: `false` (default) is a one-way claim; `true` is a validator-confirmed two-way assertion. Producers SHOULD leave the field absent on emission and let validators determine status; producers MUST NOT set `verified: true` without grounds.

For the seven schema.org-native edge types (§2.1), the cross-document verification check operates straightforwardly: the validator fetches the counter-party citemap and looks for the inverse property either inline (e.g. `subOrganization` on the parent's brand object) or as a relationship row of the inverse type.

For the six CiteMaps-extension edge types (§2.2), verification semantics vary:

- `franchiseOf` — the franchisor's citemap SHOULD emit a corresponding row for each franchisee. In practice few franchisors publish full citemaps for their franchise networks; most `franchiseOf` edges stay unverified.
- `locationOf` — when a location operates with its own citemap (rare in v3.3, more common as the spec adopts), the parent's `locationOf` edge to the child is verified by the child's reciprocal pointer at the parent's `@id`. Single-citemap producers (parent only) leave locations as `@graph` nodes; the edge stays unverified.
- `practitionerAt` — analogous to `worksFor` from the Person's perspective. A practitioner publishing their own citemap can verify by emitting the corresponding `worksFor` edge.
- `predecessorOf` / `successorOf` — verifying typically requires both predecessor and successor entities to publish, which is unusual for acquired/dissolved entities. Most stay unverified.
- `operatesAs` — self-edges are intrinsically self-verifying. Validators MAY set `verified: true` automatically on `operatesAs` self-edges that pass orphan and structural checks.
- `sellsAt` — verification would require the marketplace's own structured data to assert the inverse seller relationship. Currently unavailable from major marketplaces (Amazon, eBay, Etsy, Walmart). These stay unverified in practice.

### 2.6 Validator behavior for unknown and custom edge types

v3.3 uses open-world vocabulary semantics. Validators MUST accept relationships whose `type` is a custom URI or an unrecognized value. The intent is to keep adoption frictionless and the vocabulary evolvable.

Validator behavior on unknown types:

- Edges with `type` matching a known v3.3 taxonomy entry validate normally.
- Edges with `type` matching a known custom URI (registered with the validator) validate against the custom vocabulary's rules.
- Edges with `type` matching an unknown string or URI SHOULD produce an advisory warning ("unrecognized edge type — see §2.3 for custom type registration"). Validators MUST NOT reject the document.

Future spec versions MAY graduate custom types into the canonical registry. When that happens, the v3.3 emitter's documents continue to validate cleanly under the newer spec — the type was always valid; only the spec's documentation of it changed.

Producers using custom types in production SHOULD set the type to an absolute URL pointing at their own vocabulary documentation. This convention lets external tools dereference the URL to learn the semantics, and matches the schema.org type-resolution pattern.

## 3. Foundational sections (always-on contract)

The 10 sections every v3.3 citemap.json emits regardless of vertical selection. Replaces v3.0's flat "modules" model where these sections were optionally toggleable. v3.3 producers MUST emit all 10 foundational sections; consumers MAY rely on their presence (though they may carry empty arrays or null values where the entity has no data).

| Section | Source | Status |
|---|---|---|
| `brand` | Producer's entity identity card | always-on in v3.2; unchanged |
| `citationContract` | Auto-generated per regen | always-on in v3.2; unchanged |
| `temporalRecord` | Producer's events / timeline data | empty array as explicit "no claims" signal |
| `trust.perField` | Computed from field provenance + NAP corroboration | always-on in v3.3 |
| `answerContent` | Producer's authored Q&A (or AI-suggested fallback) | always-on in v3.3 |
| `brand.team` | Producer's team roster (see §4.5) | always-on in v3.3 |
| `brand.products` | Producer's product catalog (see §4.7) | always-on in v3.3 |
| `brand.sameAs` | Verified social profiles + storefronts auto-fold (see §4.3) | always-on in v3.3 |
| **`policy`** | Producer's editorial / correction / disclosure / privacy URLs | **NEW in v3.3** |
| **`verification`** | Producer's external verifier claims | **Promoted from optional module to foundational in v3.3** |

### 3.1 Why always-on

Three reasons the v3.3 foundational set drops the optionality v3.0's module model offered:

**Mental-model alignment.** The v3.0 module-picker UX asked users to decide which structural sections their citemap should include. In practice, sections like `temporalRecord` (entity timeline), `trust.perField` (per-claim trust scoring), and `answerContent` (authored Q&A for AI) are equally valuable for every entity type; presenting them as opt-in created decision friction with no upside. v3.3 surfaces verticals (the things that legitimately vary by entity type) and treats foundational sections as the baseline contract.

**AI engine trust.** Foundational sections are the structural signal AI engines weight most heavily when assessing the entity's published authority. A citemap.json that omits `trust.perField` because the producer chose not to emit it is indistinguishable from a citemap that omits it because the producer can't compute trust scores. v3.3's always-on contract removes that ambiguity: an empty `trust.perField` MEANS "no claims have provenance data attached," not "the producer doesn't support trust scoring."

**Competitive moat for the format.** Bare schema.org Organization markup looks structurally thin next to a v3.3 citemap carrying 10 foundational sections + a graph extension + vertical-specific module data on every emission. The always-on guarantee makes the format pitchable as "structured data, plus everything you'd want structured around it." Producers that emit valid v3.3 documents inherit that differentiation automatically.

### 3.2 Per-section schema definitions

Sections previously documented elsewhere are referenced rather than redefined. New schemas (policy, verification) are defined here. `trust.perField` is sketched here for the first time in published spec text.

|---|---|

| `citationContract` | v3.0 spec |
| `temporalRecord` | v3.0 spec |
| `answerContent` | v3.0 spec |
| `brand.team` | §4.5 above |

**`trust.perField` — schema sketch.**

```jsonc
{
  "trust": {
    "perField": {
      "name":    { "tier": 1, "sources": ["self", "google-business-profile"], "lastVerifiedAt": "2026-05-31" },
      "address": { "tier": 2, "sources": ["self", "yelp"], "lastVerifiedAt": "2026-05-31" },
      "phone":   { "tier": 1, "sources": ["self", "google-business-profile", "yelp"], "lastVerifiedAt": "2026-05-31" }
    }
  }
}
```

Per-field entries describe the trust tier (1 = highest authority — self-source + first-party verifier; 4 = lowest — self-source only) and the sources corroborating the claim. v3.3 producers SHOULD populate `trust.perField` for every entity-level field whose value appears in third-party listings (name, address, phone, email, hours). Computed by the producer; not an assertion the consumer can independently verify (the validator does not cross-check).

**`policy` — schema.**

```jsonc
{
  "policy": {
    "editorial":  { "url": "https://example.com/editorial-policy",  "summary": "Editorial independence and conflict-of-interest disclosure." },
    "correction": { "url": "https://example.com/corrections",       "policy": "Errors corrected within 24 hours with prominent correction notice." },
    "disclosure": { "url": "https://example.com/disclosures",       "summary": "Affiliate relationships disclosed inline." },
    "privacy":    { "url": "https://example.com/privacy" },
    "lastUpdated": "2026-05-16"
  }
}
```

Each sub-key is OPTIONAL. Entities without published editorial / correction / disclosure policies emit empty objects or omit the keys entirely. The `lastUpdated` field at the root SHOULD reflect the most recent change to any sub-policy. Schema.org mapping where applicable: [`Organization.publishingPrinciples`](https://schema.org/publishingPrinciples) (editorial), [`Organization.correctionsPolicy`](https://schema.org/correctionsPolicy) (correction).

**Why publisher trust as a foundational section.** AI engines and search systems already weight publisher-trust signals (NewsGuard, EEAT, the Knowledge Graph's source assessment). v3.3 surfaces them as structured fields rather than free-text prose buried in About pages. Entities with no policies in place SHOULD still emit the empty-object form; the explicit absence is itself a signal to AI engines about trust posture.

**`verification` — schema.**

```jsonc
{
  "verification": {
    "claims": [
      {
        "id":         "v_as9100d",
        "claim":      "AS9100D certified",
        "issuer":     "Aerospace Quality Management Systems",
        "issuedDate": "2024-03-15",
        "expiresAt":  "2027-03-15",
        "verifierUrl": "https://verifier.example.com/lookup?id=AS9-123456",
        "evidence":   "https://example.com/certifications/as9100d-cert.pdf"
      }
    ]
  }
}
```

**Required fields per claim:** `id`, `claim`, `issuer`. All others OPTIONAL.

Each claim represents an external-verifier assertion the entity carries (certifications, licenses, accreditations, audits, awards with verifiable issuers). v3.3 promotes `verification` from optional v3.0 module to foundational because every entity type legitimately has verification claims worth surfacing (a SaaS company's SOC 2, a healthcare practice's accreditation, a contractor's bonded-and-insured status, a publisher's source-credibility certifications). Empty `claims: []` is the explicit signal that the entity carries no third-party-issued verifications.

Verification claims feed `trust.perField` as a Tier-1 promoter: a `brand.legalName` field corroborated by a `verification.claims[]` entry referencing an external registry attestation gets Tier-1 trust regardless of whether other corroborating sources exist.

### 3.3 Empty-emission rules

Foundational sections MUST emit even when empty. Emit-as-empty (e.g. `"team": []`, `"verification": { "claims": [] }`) is structurally honest:

- **Empty-array vs missing-field distinguishes "no data" from "not implemented".** A v3.3 producer that omits `brand.team` entirely tells consumers "this producer doesn't know about team data." A v3.3 producer that emits `"team": []` tells consumers "this entity has no public-facing team members." Different semantic; both have downstream consequences.
- **AI engines reason about absence.** An entity emitting `"verification": { "claims": [] }` is signaling "we carry no third-party verifications" — a real signal AI engines weight when assessing authority. Omission deprives the AI of that signal.
- **Empty-array shapes are cheap.** Bytes are not the constraint; semantic clarity is.

The single exception: `trust.perField` MAY be omitted if the producer has no provenance data for any field (typical for v3.3-conformant producers operating below the trust-scoring data threshold). Producers SHOULD emit at least `{}` to indicate the section exists but carries no data.

### 3.4 Foundational-vs-vertical decision matrix

Producers adding new fields in future v3.x versions can determine whether a field belongs in the foundational always-on contract or as a vertical-scoped emission using three questions:

1. **Is the field semantically meaningful for entities of all verticals?** If yes → foundational candidate. If no → vertical-scoped.
2. **Would the field's absence be misinterpretable as "not implemented" by consumers?** If yes → foundational (so empty-emission signals "no data" unambiguously). If no → vertical-scoped is fine.
3. **Does the field describe the entity's identity / trust / authorial output / network presence?** If yes → foundational (these are the categories the v3.3 always-on set covers). If it describes operational specifics of a particular industry → vertical-scoped.

A field that passes all three SHOULD be foundational. A field that fails any one SHOULD be vertical-scoped. Borderline cases (e.g. `accreditations` — relevant to many verticals but not all) MAY be vertical-scoped with cross-references in the spec text noting the cross-vertical applicability.

## 4. Constituent field additions

Fields that have been emitted by Studio's reference implementation ahead of the spec, formalized in v3.3. Each subsection documents the canonical shape, semantics, derivation rules from upstream producer data, and the schema.org mapping where relevant. All fields are OPTIONAL on the producer side; consumers MAY rely on their semantics when present but MUST NOT require them.

Sub-field signatures shown in the schema blocks are illustrative; the canonical structure is defined in the [v3.3 JSON Schema](https://citemaps.org/schema/v3.3/citemap.schema.json).

### 4.1 `brand.legalName`

The full legal-entity name of the primary entity (e.g. "Western Filament, Inc.") as distinct from `brand.name` (the display or common name, e.g. "Western Filament").

**Schema:**

```jsonc
{
  "brand": {
    "name": "Western Filament",
    "legalName": "Western Filament, Inc."
  }
}
```

**Schema.org mapping:** [`Organization.legalName`](https://schema.org/legalName).

**Semantics.** `brand.name` remains the canonical display name and is REQUIRED. `brand.legalName` is OPTIONAL and used as a soft-match anchor for NAP and social-profile drift comparators: when an external listing carries the legal variant ("Western Filament, Inc.") instead of the display name ("Western Filament"), comparators that recognize both values do not produce false-positive drift reports. Producers SHOULD set `legalName` for incorporated entities whose legal form (Inc / LLC / Pty Ltd / GmbH / SARL / etc.) commonly appears on third-party listings.

### 4.2 `brand.email`

A public-facing business email for the primary entity. NAP-adjacent in role — not part of the strict Name/Address/Phone trinity, but AI engines reading directory listings frequently surface email addresses alongside the other three.

**Schema:**

```jsonc
{
  "brand": {
    "name": "Acme HVAC",
    "email": "hello@acmehvac.com"
  }
}
```

**Schema.org mapping:** [`Organization.email`](https://schema.org/email) and [`LocalBusiness.email`](https://schema.org/email). Producers SHOULD emit at the top-level entity AND duplicate as `contactPoint[0].email`; Google's Knowledge Graph reads the top-level form, other consumers walk the `contactPoint` array.

**Semantics.** Distinct from per-team-member email (which lives at `brand.team[*].email`, §4.5). The `brand.email` is the entity's published general-purpose contact address — the address a customer would email if the website's contact form is broken. Producers MUST NOT set `brand.email` to a personal employee address; that semantic belongs in `brand.team[*].email`.

### 4.3 `brand.sameAs[]`

Identity-equivalent URLs that point at the primary entity's other canonical representations on the web — verified social profiles, Wikipedia, Wikidata, Crunchbase, marketplace storefronts, and similar.

**Schema:**

```jsonc
{
  "brand": {
    "sameAs": [
      "https://www.linkedin.com/company/acmehvac",
      "https://www.facebook.com/acmehvac",
      "https://www.amazon.com/stores/acmehvac",
      "https://en.wikipedia.org/wiki/Acme_HVAC"
    ]
  }
}
```

**Schema.org mapping:** [`Organization.sameAs`](https://schema.org/sameAs).

**Derivation.** v3.3 producers derive `brand.sameAs[]` from two upstream sources unified into a single output array:

1. Verified social-profile listings — only profiles that have passed identity verification (NAP match, claim flow, or owner attestation) are emitted. Unverified or speculative profile claims SHOULD NOT appear.
2. Marketplace storefronts emitted via `brand.storefronts[]` (§4.12) — each storefront URL auto-folds into `sameAs[]` so consumers that read only `sameAs` see the entity's marketplace presence as identity-equivalent.

Order is not semantically significant. Duplicates MUST be deduplicated.

### 4.4 `brand.taxonomy` revisions

v3.3 retains `brand.taxonomy` from v3.2 with two changes driven by the §6 vertical-taxonomy retirements:

- Four values are deprecated from the recommended `primaryVertical` registry (`local-business`, `manufacturer`, `membership-organization`, `industrial-services-b2b`). Each remains a valid string for back-compat; the registry no longer recommends them.
- The vocabulary registry version bumps from `2026-05` to `2026-Q3` (or the current registry release at v3.3 ship time) to surface the change to consumers via the `taxonomyVersion` field.

All other `brand.taxonomy` semantics (primaryVertical / additionalVerticals / subtypes / entityType / organizationType / modifiers — see v3.2 §1) carry forward unchanged. See §6 for the retirement rationale and migration semantics.

### 4.5 `brand.team[]`

A roster of public-facing team members associated with the primary entity — providers, partners, named agents, executives. Each entry describes one person; consumers MAY use the roster to populate practitioner-network queries, surface named experts, or graduate the entries into the `@graph[]` layer as Person nodes (§1) connected via `worksFor` or `practitionerAt` edges (§2).

**Schema:**

```jsonc
{
  "brand": {
    "team": [
      {
        "id": "tm_jane_doe",
        "name": "Jane Doe",
        "honorificPrefix": "Dr.",
        "honorificSuffix": "MD, PhD",
        "jobTitle": "Chief Medical Officer",
        "bio": "Board-certified in…",
        "photoUrl": "https://example.com/jane.jpg",
        "email": "jane@example.com",
        "phone": "+1-555-0100",
        "sameAs": ["https://linkedin.com/in/janedoe"],
        "degrees": [
          { "degree": "MD", "field": "Internal Medicine", "institution": "Stanford University School of Medicine", "year": 2008 }
        ],
        "training": [
          { "type": "Residency", "program": "Internal Medicine", "institution": "Mayo Clinic", "year": 2011 }
        ],
        "priorPositions": [
          { "title": "Attending Physician", "organization": "Mayo Clinic", "startYear": 2011, "endYear": 2016 }
        ],
        "publications": [
          { "title": "Outcomes of…", "venue": "JAMA", "year": 2019, "url": "https://…" }
        ],
        "achievements": ["Castle Connolly Top Doctor 2023"],
        "credentials": {
          "healthcare": { "npi": "1234567890", "boardCertifications": ["ABIM"] }
        },
        "formerlySince": null,
        "formerlyReason": null
      }
    ]
  }
}
```

**Schema.org mapping.** Each entry maps to a [`Person`](https://schema.org/Person) node with the following property correspondences:

| Team field | Schema.org property |
|---|---|
| `name`, `honorificPrefix`, `honorificSuffix`, `jobTitle`, `description` (bio), `image` (photoUrl), `email`, `telephone`, `sameAs` | Direct property equivalents on `Person` |
| `degrees[]`, `training[]` | [`hasCredential`](https://schema.org/hasCredential) as `EducationalOccupationalCredential` (Degree or Certification) with `recognizedBy` `Organization` stubs |
| `priorPositions[]` | [`alumniOf`](https://schema.org/alumniOf) `Organization` stubs |
| `achievements[]` | [`award`](https://schema.org/award) entries (one per string) |
| `credentials.{vertical}.{idType}` (NPI / barNumber / CRD / licenseNumber) | [`identifier`](https://schema.org/identifier) as `PropertyValue` with `propertyID` set to the credential type |

**Inactive (departed) members.** Members who have left the entity MAY remain in `brand.team[]` with `formerlySince` (ISO date) and an optional `formerlyReason` set. Inactive members:

- MUST be excluded from the entity's `Organization.employee[]` schema.org emission (employee list reflects current staff only)
- SHOULD remain in `brand.team[]` for AEO continuity (third-party listings, reviews, and citations attached to the person's name still resolve)
- MAY surface in `@graph[]` as a Person node connected via a `worksFor` edge whose `observedAt` reflects the historical tenure

**Graduation to `@graph` Person nodes.** v3.3 producers MAY emit team members as `@graph[]` Person nodes with `worksFor` or `practitionerAt` edges (§2.1 / §2.2) connecting them to the primary entity. This is the canonical v3.3 representation for practitioner-network entities (healthcare practices, legal firms, professional services). Producers without graph-extension support continue to emit the denormalized `brand.team[]` form; consumers MUST be tolerant of both.

### 4.6 `brand.teamPageUrl`

Canonical URL where the entity publishes its team or practitioner roster (e.g. `/about`, `/team`, `/providers`).

**Schema:**

```jsonc
{
  "brand": {
    "teamPageUrl": "https://example.com/team"
  }
}
```

**Semantics.** Used by AEO tooling as a Tier 0 self-source target — the entity's own canonical page about its team is the highest-trust source for team-member identity verification, ahead of third-party directory listings. AI engines that follow the URL can confirm whether a person referenced in `brand.team[]` is actually surfaced on the entity's own site.

### 4.7 `brand.products[]`

A catalog of products the primary entity offers, structured per-product with rich metadata. Distinct from per-product-page schema.org `Product` emission — `brand.products[]` is the entity-level catalog that powers AI-shopping discovery, comparison queries, and price-range surfacing.

```jsonc
        "pageUrl": "https://example.com/products/aerospace-lacing-tape",

        "availability": "in-stock",

          "amazon": "https://amazon.com/dp/B0...",

          { "name": "Brand X Lacing Tape", "url": "https://..." }

        "productForm": "physical",
```

**Required fields per entry:** `id`, `name`. All others OPTIONAL.

**Schema.org mapping.** Each entry maps to a [`Product`](https://schema.org/Product) node. The per-product `brand` string MAY reference a `brand.subBrands[]` entry by name (§4.9); when matched, schema.org emission enriches the product's `brand` property with the full `Brand` node (logo, url, sameAs, description).

**Foundational status.** `brand.products[]` is one of the §3 always-on foundational sections. v3.3 producers MUST emit the field even when empty (`"products": []` — an explicit "no catalog" signal). When the upstream producer has no product data, AI-suggested fallbacks MAY populate the array; producers using AI fallbacks SHOULD set the per-entry `source` field to indicate provenance.

The entity-level catalog is canonical; module-level duplication is rejected to prevent the double-emission bug surfaced during pre-v3.3 dogfood.

### 4.8 `brand.locations[]`

A catalog of physical locations the primary entity operates from. Pairs with the `locationOf` edge type (§2.2) for v3.3 graph-extension consumers.

**Schema:**

```jsonc
{
  "brand": {
    "locations": [
      {
        "id": "loc_bend_hq",
        "type": "headquarters",
        "name": "Bend Headquarters",
        "address": "1234 Industrial Way",
        "city": "Bend",
        "region": "OR",
        "postalCode": "97703",
        "country": "US",
        "phone": "+1-541-555-0100",
        "email": "bend@example.com",
        "hours": "Mon-Fri 8am-5pm",
        "url": "https://example.com/locations/bend",
        "googlePlaceId": "ChIJ...",
        "isPrimary": true
      }
    ]
  }
}
```

**Required fields per entry:** `id`, `type`. All others OPTIONAL.

**`type` enum:** `headquarters`, `production`, `office`, `retail`, `warehouse`, `showroom`, `popup`, `regional`, `international`, `other`.

**Single-primary semantics.** At most one entry MAY have `isPrimary: true`. Producers with multiple `isPrimary: true` rows MUST resolve to a single primary at emission time (first-seen wins is a reasonable default).

**Empty-catalog semantics.** An empty `brand.locations[]` MEANS the entity operates from a single location described by the entity's existing `brand.location` / `brand.address` fields. Producers MUST NOT emit `brand.locations[]` as a single-entry array duplicating the entity's headline location; that's the empty-catalog case.

**Operational-independence threshold.** Locations that meet multiple operational-independence criteria (separate NAP, separate Google Business Profile, separate citation surface, separate monitoring needs) SHOULD be represented as separate primary entities with their own citemap.json and connected via the `locationOf` edge (§2.2) rather than as catalog entries. Multi-location dental practices, for instance, typically warrant per-location citemaps once each office runs its own GBP. Catalog entries are appropriate for locations sharing the parent entity's citation surface.

### 4.9 `brand.subBrands[]`

A catalog of sub-brand identities the primary entity operates under. Distinct from legal subsidiaries (which warrant their own citemap connected via `subOrganizationOf`, §2.1) — `brand.subBrands[]` is for marketing lines, product brands, and DBA names that share the parent's legal and AEO identity.

**Schema:**

```jsonc
{
  "brand": {
    "subBrands": [
      {
        "id": "sb_premium_line",
        "name": "Acme Premium",

      }
    ]
  }
}
```

**Required fields per entry:** `id`, `name`. All others OPTIONAL.

**Single-primary semantics.** Same rule as §4.8 — at most one `isPrimary: true` row. The primary sub-brand is the de facto default identity used when the entity's primary `brand.name` is genericized.

**Empty-catalog semantics.** An empty `brand.subBrands[]` MEANS the entity operates under a single brand identity (`brand.name`). Producers MUST NOT emit a single-entry duplicate.

**Per-product brand enrichment.** When a `brand.products[*].brand` string matches a `brand.subBrands[*].name`, v3.3 consumers MAY enrich the per-product schema.org `Brand` node with the sub-brand's logo, url, sameAs, and description. This lets consumers surface accurate sub-brand attribution for products without producers having to duplicate brand metadata per product.


**Required fields per entry:** `id`, `name`. All others OPTIONAL.

**Nesting cap.** v3.3 enforces a 3-level nesting cap on category trees. Categories whose `parentId` chain depth exceeds 3 MUST be flattened to root on emission. Producers MUST detect and break cycles (a category whose parentId chain includes itself).

**Category page mapping.** When `pageUrl` is set on a category, schema.org `CollectionPage` + `ItemList` emission for that category page lists the products / services with matching `categoryIds[]`. v3.3 producers MAY emit per-category-page schema.org artifacts alongside the citemap.

### 4.11 `brand.services[]`

A catalog of services the primary entity offers. Structural parallel to `brand.products[]` (§4.7) for service-centric entities.

**Schema:**

```jsonc
{
  "brand": {
    "services": [
      {
        "id": "s_consulting",
        "name": "Strategic Consulting",
        "serviceType": "advisory",
        "description": "C-suite advisory engagements for enterprise transformation.",
        "pageUrl": "https://example.com/services/strategic-consulting",
        "priceRange": "$$$",
        "pricingModel": "retainer",
        "areaServed": "North America",
        "audience": "Enterprise leadership",
        "duration": "6-12 months",
        "deliveryModel": "hybrid",
        "bookingUrl": "https://example.com/book/consulting",
        "inquiryUrl": "https://example.com/contact",
        "brand": "Acme Premium",
        "categoryIds": ["ct_advisory"],
        "isPrimary": false,
        "featured": true
      }
    ]
  }
}
```

**Required fields per entry:** `id`, `name`. All others OPTIONAL.

**`pricingModel` enum:** `hourly`, `flat-fee`, `retainer`, `project-based`, `subscription`, `commission`, `contingency`, `free-consult`, `quote-only`. Consumers MUST treat unknown values as `quote-only` (the most conservative interpretation).

**`deliveryModel` enum:** `in-person`, `remote`, `hybrid`, `phone`, `online-only`, `appointment-required`. Same unknown-value rule as `pricingModel`.

**Schema.org mapping.** Each entry maps to a [`Service`](https://schema.org/Service) node, with `pricingModel` and `deliveryModel` surfaced via [`offers`](https://schema.org/offers) and [`availableChannel`](https://schema.org/availableChannel) respectively.

**Single-primary semantics.** At most one `isPrimary: true` row, same as §4.8 and §4.9. Suppression rule applies symmetric to §4.7: when `brand.services[]` is populated, v3.3 producers MUST NOT also emit per-vertical service arrays (`services`, `serviceList`, `serviceOfferings` at module level).

### 4.12 `brand.storefronts[]`

The entity's catalog of marketplace storefronts. Entity-level marketplace presence — distinct from per-product marketplace listings which live at `brand.products[*].externalListings`. Pairs with the `sellsAt` edge type (§2.2) for graph-extension consumers.

**Schema:**

```jsonc
{
  "brand": {
    "storefronts": [
      {
        "id": "sf_amazon",
        "platform": "amazon",
        "url": "https://www.amazon.com/stores/acmehvac",
        "displayName": "Acme HVAC on Amazon"
      },
      {
        "id": "sf_custom",
        "platform": "custom",
        "customPlatform": "Bend Local Co-op",
        "url": "https://bendcoop.com/shop/acme",
        "displayName": "Acme at Bend Co-op"
      }
    ]
  }
}
```

**Required fields per entry:** `id`, `platform`, `url`. All others OPTIONAL.

**`platform` enum:** `amazon`, `ebay`, `etsy`, `walmart`, `faire`, `shopify`, `google-shopping`, `wayfair`, `reverb`, `apple-app-store`, `google-play`, `custom`. Entries with `platform: "custom"` MUST set `customPlatform` (free-text platform name).

**Derivation interaction.** Storefront URLs auto-fold into `brand.sameAs[]` (§4.3) so consumers reading only `sameAs` see the entity's marketplace presence. Producers MUST NOT manually duplicate storefront URLs in `sameAs[]`; the auto-fold handles it.

**Graph extension.** v3.3 graph-extension producers MAY emit each storefront as a `sellsAt` edge from the entity to a marketplace node in `@graph[]`. The edge stays `verified: false` until the marketplace publishes structured data confirming the inverse seller relationship — currently unavailable from major marketplaces.

### 4.13 `brand.publications[]`

A catalog of structured published works associated with the primary entity. Pairs with the §3 foundational `expertise[]` topic registry: `publications[]` carries concrete works (articles, case studies, press releases, podcasts, videos, testimonials, awards); `expertise[]` carries topical claims derived from or independent of those works.

**Schema:**

```jsonc
{
  "brand": {
    "publications": [
      {
        "id": "pub_2024_industry_award",
        "title": "Acme HVAC Wins Industry Innovation Award",
        "url": "https://industryjournal.com/2024-awards",
        "contentType": "award",
        "publisher": "Industry Journal",
        "publishedAt": "2024-03-15",
        "description": "Recognized for breakthrough refrigerant recovery technology.",
        "authorName": "Industry Journal Staff",
        "authorTeamMemberId": null,
        "role": "featured"
      }
    ]
  }
}
```

**Required fields per entry:** `id`, `title`, `url`, `contentType`. All others OPTIONAL.

**`contentType` enum:** `article`, `case-study`, `white-paper`, `press-release`, `podcast`, `video`, `testimonial`, `award`.

**`role` enum:** `authored` (the entity or a team member wrote the work), `featured` (the work is about the entity), `mentioned` (the work references the entity in passing). When omitted, role defaults are per-contentType: `testimonial` and `award` default to `featured` (the entity is the subject by definition); all others default to `authored`.

**Schema.org mapping.** Per-publication schema.org emission varies by contentType and role:

| contentType | Default schema.org type | Role-aware property |
|---|---|---|
| `article`, `case-study`, `white-paper` | [`Article`](https://schema.org/Article) | `authored` → `creator` / `author`; `featured` → `about`; `mentioned` → `mentions` |
| `press-release` | [`NewsArticle`](https://schema.org/NewsArticle) | Same role mapping |
| `podcast` | [`PodcastEpisode`](https://schema.org/PodcastEpisode) | Same role mapping |
| `video` | [`VideoObject`](https://schema.org/VideoObject) | Same role mapping |
| `testimonial` | [`Review`](https://schema.org/Review) | `author` = reviewer; `itemReviewed` = entity (regardless of role) |
| `award` | [`CreativeWork`](https://schema.org/CreativeWork) with `award` | `authored` → `author` = team member; otherwise `about` = entity |

**Team-member author linking.** When `authorTeamMemberId` references a `brand.team[*].id`, the schema.org `author` / `creator` property emits as the linked Person node with that team member's `name`, `jobTitle`, and `profileUrl`. When the referenced member has `formerlySince` set (departed team), the schema.org emission notes the historical authorship without implying current employment.

### 4.14 Promoted-from-vertical fields (no new fields, canonicalization)

v3.3 does not introduce new spec fields for the nine cross-vertical attributes promoted to entity-top-level in Studio's reference implementation. The change is purely about canonical-source location: properties with multiple verticals that previously exposed duplicate input surfaces for the same conceptual field (e.g. `certifications` shared between `manufacturer` and `industrial-services`) now have a single source-of-truth.

The promoted fields:

`certifications`, `industriesServed`, `licenseNumber`, `bonded`, `insured`, `emergencyAvailable`, `bookingUrl`, `feeStructure`, `insurancesAccepted`.

**Emission paths unchanged.** v3.3 consumers continue to read these from their per-vertical module paths (`legal.fee.model`, `finance.certifications`, `realEstate.brokerLicense.license`, `healthcare.insurance`, etc.). The promotion is internal to producers; the spec's emission surface is unchanged.

**Divergent-semantic fields stay per-vertical.** Fields that look like they could promote but actually carry different semantics across verticals (`serviceTypes`, `categories`, `amenities`, `projectTypes`, `ageGroups`, `focusAreas`, `practiceAreas`) explicitly do not promote. Each vertical's interpretation of these fields is distinct enough that flattening to a top-level field would lose meaning. Producers continue to emit these from per-vertical sources.

## 5. Modules → verticals reframe

Reorganizes v3.0's flat "modules" catalog into 14 user-pickable verticals plus the 10 foundational sections from §3. The v3.0 catalog conflated three different concepts under one flat list: true verticals (taxonomic classifications), foundational sections (universally-applicable structural blocks), and operational modifiers (already lifted to `Property.businessModel[]` in v3.2). v3.3 separates them cleanly.

The reframe touches three surfaces that consume the module catalog: public citemap.json producers (the citemaps.org generator surface), schema-generation tooling, and AI-side parsers that classify entities by module key. Each gets a cleaner mental model — verticals match how users describe their business; foundational sections describe the always-on baseline; operational modifiers describe how the entity operates rather than what it is.

### 5.1 Mapping table: v3.0 module key → v3.3 outcome

| v3.0 module key | v3.3 outcome | Notes |
|---|---|---|
| `local-business` | **DEPRECATED** as vertical; remains valid `siteType` | Retired from primaryVertical registry per §6. Existing producers continue to validate; user-picker no longer offers it. |
| `ecommerce` | vertical:`ecommerce` | unchanged |
| `saas` / `software` | vertical:`saas` | Alias unification — both v3.0 keys map to the same v3.3 vertical |
| `content` | vertical:`content` | unchanged |
| `events` | vertical:`events-venue` | renamed (see §5.2) |
| `real-estate` | vertical:`real-estate` | unchanged |
| `education` | vertical:`education` | unchanged |
| `health` | vertical:`healthcare` | renamed (see §5.2) |
| `legal` | vertical:`legal-services` | renamed (see §5.2) |
| `finance` | vertical:`financial-services` | renamed (see §5.2) |
| `nonprofit` | vertical:`nonprofit` | unchanged |
| `creative` | vertical:`creative-artist` | renamed for scope clarity |
| `government` | vertical:`government` | unchanged |
| `science` | vertical:`science-research` | renamed for scope clarity |
| (new in v3.3) | vertical:`restaurant` | Promoted from local-business subtype to first-class vertical |
| `brand` (foundational) | **foundational** | always-on per §3 |
| `citationContract` (foundational) | **foundational** | always-on per §3 |
| `temporalRecord` (foundational) | **foundational** | always-on per §3 |
| `answerContent` (foundational) | **foundational** | always-on per §3 |
| `verification` (optional module) | **foundational** | promoted per §3 |
| `people` | **DROPPED** | superseded by `brand.team[]` per §4.5 |
| `places` | **DROPPED** | superseded by `brand.location` + `brand.locations[]` per §4.8 |
| `business-IP` | **DROPPED** | consolidated into `brand` (logo, founded, legalName, etc. already there) |
| `businessModel` modifiers | **DROPPED as module** | already moved to `Property.businessModel[]` in v3.2; emits under `brand.taxonomy.modifiers[]` |

Final count: **14 user-pickable verticals** + **10 always-on foundational sections** + 4 dropped legacy modules.

### 5.2 Renames

Four v3.0 module keys rename in v3.3 to match common usage and clarify scope. Each rename emits both keys during the dual-write window (§5.5).

| v3.0 key | v3.3 key | Why |
|---|---|---|
| `events` | `events-venue` | Distinguishes venues-that-host-events (operatic, sports arenas, conference centers) from event-organizer entities. v3.4 may introduce a separate `events-organizer` vertical. |
| `health` | `healthcare` | Common-usage form across consumer-facing and regulatory contexts. Aligns with schema.org's `MedicalBusiness` ecosystem. |
| `legal` | `legal-services` | `legal` as a bare adjective is ambiguous (could mean "lawful" generally). Specifies the service-provider category. |
| `finance` | `financial-services` | Same scope-clarity rationale as `legal-services`. Distinguishes service-provider entities from coverage of financial topics (which would be `content` with finance subtopics). |
| `creative` | `creative-artist` | v3.0's `creative` was used for both creative agencies and individual creators; v3.2's existing `creative-artist` siteType captures the individual-creator semantic. v3.4 may add `creative-agency` as a sibling. |
| `science` | `science-research` | Distinguishes research-doing entities (labs, institutes) from science-publishing entities (which would be `content` with science subtopics). |
| `saas` / `software` | `saas` | Alias unification — `software` was historically used inconsistently for both SaaS products and on-premise software vendors. v3.3 collapses both to `saas` until a real `software-on-premise` use case emerges. |

### 5.3 New verticals promoted from subtypes

`restaurant` graduates from a subtype of `local-business` to a first-class vertical in v3.3. The promotion recognizes that restaurants have distinct AEO playbook depth: menu-specific structured data, reservation systems, delivery-platform integrations, dietary-restriction signaling, hours-of-operation specificity, food-safety certifications — none of which apply to local businesses generically. Restaurants currently mapping to `local-business` SHOULD migrate to `restaurant` on next regen.

No other subtypes promote to vertical status in v3.3. Future versions MAY promote based on observed usage patterns (e.g. `home-services`, `automotive` could be candidates if dogfood reveals that the local-business catchall obscures playbook depth).

### 5.4 Drops

Three v3.0 modules drop in v3.3 because their data has migrated to better-fitting surfaces:

| v3.0 module | Why dropped | Replacement |
|---|---|---|
| `people` | The module was originally meant for organizations whose primary subject is a person (a personal brand, a public figure). In practice `brand.team[]` (§4.5) covers the named-people surface for organizational entities; `brand.taxonomy.entityType: "person"` covers the personal-brand case more cleanly than a separate module. | `brand.team[]` for organizational team rosters; `brand.taxonomy.entityType: "person"` for personal-brand entities |
| `places` | Originally for entities whose primary subject is a location (a city, a landmark, a destination). The use case is real but small; `brand.taxonomy.entityType: "place"` plus `brand.locations[]` (§4.8) covers it. The standalone module added complexity without unique value. | `brand.taxonomy.entityType: "place"` + `brand.locations[]` |
| `business-IP` | Originally for intellectual-property attributes (logo, founded date, legal name, registered trademarks). Every one of these fields already lives at `brand.*` top-level (`brand.logo`, `brand.founded`, `brand.legalName`, etc.). The module was a redundant container. | Direct `brand.*` fields |

Producers emitting any of these as v3.0 modules SHOULD migrate the data to the replacement surface on next regen. v3.3 consumers that encounter legacy `people` / `places` / `business-IP` modules SHOULD parse them charitably (extract whatever fields they can) and emit a deprecation warning.

### 5.5 Legacy-key dual-write window

v3.3 producers emit both the old and new key names during a one-release-version transition window. The dual-write applies to:

- The four renames in §5.2 (`events` ↔ `events-venue`, `health` ↔ `healthcare`, `legal` ↔ `legal-services`, `finance` ↔ `financial-services`, `creative` ↔ `creative-artist`, `science` ↔ `science-research`, `software` → `saas`)
- The vertical-vs-foundational re-classification of `verification` (emit both as a top-level module AND under the foundational `verification` section)

Dual-write does NOT apply to:

- The four DROPPED modules (`people`, `places`, `business-IP`, legacy `businessModel`). Producers MUST stop emitting these in v3.3; the replacement surfaces are authoritative.
- The DEPRECATED `local-business` vertical (handled separately per §6 — siteType enum unchanged; only primaryVertical registry deprecates).

Dual-write window: v3.3 emits both old and new. **v3.4 drops the legacy keys.** Consumers that hardcoded the v3.0 keys have one release version to migrate to the v3.3 forms. Validators MAY warn on v3.3 documents that emit ONLY the new keys (missing the legacy mirror) during the window, since older consumers will fail to find the data.

After v3.4 ships, the legacy keys MUST NOT be emitted. Consumers reading v3.4+ documents MUST use the v3.3 names.
