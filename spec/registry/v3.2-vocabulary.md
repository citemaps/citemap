# citemap.json v3.2 Recommended Values Registry

> Vocabulary for `brand.taxonomy` introduced in [v3.2](../v3.2.md). Versioned independently of the spec.

**Registry version:** 2026-05
**Spec version:** 3.2
**Status:** Living document
**License:** MIT

---

## Why a separate registry

The v3.2 spec defines the *shape* of `brand.taxonomy` — field names, types, required/optional status, validator behavior. The *vocabulary* (which verticals exist, which modifiers are common, which entity types are recognized) is a different artifact: it evolves more frequently, doesn't change the protocol, and shouldn't trigger spec version bumps.

This registry is referenced by v3.2's `brand.taxonomy.taxonomyVersion` field. Producers writing a citemap SHOULD set `taxonomyVersion` to the registry version they authored against (e.g. `"2026-05"`); consumers that recognize the version interpret values according to that registry's semantics.

Adding a new vertical or modifier triggers a registry version bump (e.g. `2026-05` → `2026-08`). Removing or renaming a value requires a deprecation cycle of at least one published version interval. Aliases let renamed values keep working: a consumer that sees `local-business` (legacy) MAY map to whatever the current registry says it became.

## `entityType`

Schema.org-style entity type. Identifies what kind of thing this citemap describes. Default: `organization`.

| Value | Description |
|---|---|
| `organization` | A business, institution, or other organized entity |
| `person` | An individual professional, public figure, or named subject |
| `place` | A geographic location, landmark, or natural area |
| `product` | A specific named product (consumer goods, software, physical or digital) |
| `service` | A specific named service offering |
| `event` | A scheduled or recurring event |
| `publication` | A periodical, blog, podcast, journal, or other publishing entity |
| `government-entity` | A government agency, department, or public institution |

### Examples for non-organization entities

The taxonomy works for entities other than companies. A few worked examples:

**Author / professional (`person`):**

```json
"taxonomy": {
  "taxonomyVersion": "2026-05",
  "primaryVertical": "publisher",
  "subtypes": ["food-writer", "memoirist"],
  "entityType": "person"
}
```

**Conference (`event`):**

```json
"taxonomy": {
  "taxonomyVersion": "2026-05",
  "primaryVertical": "events",
  "subtypes": ["industry-conference", "annual"],
  "entityType": "event",
  "modifiers": ["b2b", "experience-driven"]
}
```

**National park (`place`):**

```json
"taxonomy": {
  "taxonomyVersion": "2026-05",
  "primaryVertical": "recreation-tourism",
  "subtypes": ["national-park", "scenic-drive"],
  "entityType": "place",
  "modifiers": ["seasonal"]
}
```

These illustrate that `entityType` describes the SUBJECT of a citemap — not a product the entity sells. A SaaS company's citemap is `organization`, even though the company sells software; a citemap dedicated to a single product would have `entityType: "product"`. Producers SHOULD scope citemaps deliberately rather than emit per-product citemaps as a default.

## `organizationType`

Legal or institutional form. Captures *what kind of organization* the entity is, structurally. Operational structures (franchise, marketplace, platform) belong in `modifiers`, not here.

| Value | Description |
|---|---|
| `independent-business` | Privately held single-entity business; not part of a corporation or chain |
| `multi-location-brand` | A brand operating multiple owned locations (corporate, not franchised) |
| `corporation` | A legally incorporated entity, public or private |
| `nonprofit` | A 501(c)(3) or equivalent charitable organization |
| `government` | A government agency, department, or public-sector entity |
| `association` | A professional or industry association |
| `professional-practice` | A licensed-professional entity (law firm, medical practice, accounting firm) |
| `membership-organization` | An organization whose primary structure is dues-paying members (chamber, alumni group, club) |

### Catchalls and deprecations

The following values are RESERVED for transitional use only and SHOULD NOT be set as `primaryVertical` in new citemaps:

| Value | Status | Replacement |
|---|---|---|
| `local-business` | Deprecated as primaryVertical | Pick a specific vertical from above. The `local-business` value persists in `brand.siteType` for module-gating back-compat; producers that want to signal "physical local catchment" SHOULD include `modifiers: ["local"]` |

## `modifiers`

Cross-cutting attributes describing how the entity operates. The list below is grouped by axis for clarity but stored as a flat array in citemap files. Producers MAY use values outside this list (subject to consumer interpretation).

### Customer relationship

| Value | Description |
|---|---|
| `b2b` | Sells primarily to businesses |
| `b2c` | Sells primarily to consumers |
| `b2b2c` | Sells to businesses that resell to consumers |

### Delivery

| Value | Description |
|---|---|
| `local` | Physical local catchment; serves a specific geographic radius |
| `service-area` | Provider travels to customer; service-area definition rather than fixed location |
| `online-only` | No physical presence; delivered entirely via internet |
| `mobile-service` | Mobile/portable service that comes to the customer |

### Engagement

| Value | Description |
|---|---|
| `appointment-based` | Service requires advance scheduling |
| `walk-in` | No appointment necessary; first-come/first-served |
| `membership-based` | Recurring access pattern — gym memberships, clubs |
| `subscription` | Recurring billing with continuous service delivery |
| `rental` | Time-bounded asset use |

### Service shape

| Value | Description |
|---|---|
| `emergency-service` | 24/7 or rapid-response capability |
| `quote-based` | Pricing is custom per engagement; estimator workflow |
| `project-based` | Multi-week or multi-month engagements |
| `recurring-service` | Regular cadence (monthly cleaning, quarterly maintenance) |
| `inventory-based` | Sells from stock |
| `ecommerce-enabled` | Online checkout/cart available, even if primary is in-store |

### Trust / regulation

| Value | Description |
|---|---|
| `licensed-professional` | Requires occupational license to practice |
| `regulated` | Subject to industry or government regulation beyond standard business law |
| `ymyl` | "Your Money or Your Life" — health, finance, legal, safety subject matter |

### Positioning

| Value | Description |
|---|---|
| `luxury` | High-end, premium-priced segment |
| `budget` | Value/economy segment |
| `high-consideration` | Customer decision involves significant research or investment |

### Discovery

| Value | Description |
|---|---|
| `review-driven` | Customer choice driven primarily by reviews |
| `referral-driven` | Customer choice driven primarily by word-of-mouth |
| `lead-generation` | Sales motion centered on captured leads |
| `comparison-driven` | Customer comparison-shops across alternatives before choosing |
| `experience-driven` | Choice driven by the experience itself rather than predicted outcomes |

### Distribution

| Value | Description |
|---|---|
| `marketplace` | Operates a marketplace (third parties transact via the entity) |
| `multi-location` | Multiple physical locations under one brand |
| `franchise` | Franchised operation (franchisee or franchisor) |
| `platform` | Software/services platform that other businesses build on |

### Time

| Value | Description |
|---|---|
| `seasonal` | Operates on a non-year-round cadence |

## Versioning Policy

- **Additive changes** (new vertical, new modifier, new entityType value) trigger a registry minor version bump (e.g. `2026-05` → `2026-08`).
- **Renames** require a deprecation cycle. The old name keeps working for at least one published interval after the rename; alias entries map old to new during that window.
- **Removals** are a last resort. A removed value is documented in the registry's deprecation log for at least 12 months after removal.
- **Spec-level changes** (e.g. adding a new field to `brand.taxonomy`) are NOT registry changes; they require a v3.x spec revision.

The registry version is the second component of the producer's `taxonomy.taxonomyVersion`. A producer that wrote against `2026-05` will continue producing valid v3.2 citemaps even after the registry advances to `2026-08`; the version tag tells future consumers what semantics to apply.
