# citemap.json Specification v3.2 Addendum

> One backward-compatible addition that carries structured business taxonomy alongside `siteType`

**Status:** Draft
**Version:** 3.2
**Date:** May 2026
**Authors:** Brian Pofahl
**License:** MIT
**Schema:** https://citemaps.org/schema/v3.2/citemap.schema.json
**Backward Compatibility:** Every valid v2.0, v3.0, and v3.1 citemap validates against v3.2

---

## Overview

CiteMap v3.2 adds one feature on top of the [v3.1 specification](./v3.1.md). The addition is optional and backward-compatible — no v2.0, v3.0, or v3.1 fields are removed, renamed, or changed in meaning. A v3.1 file passes v3.2 validation without modification.

v3.2 addresses a single coordination gap surfaced through real-world deployment: **the granularity mismatch between `brand.siteType` and how producers actually classify their businesses**. The 18-value `siteType` enum is intentionally coarse — it's a module gate, not a taxonomy — but producers (Studio editors, schema generators, manual authors) often have richer structured information about the entity that gets discarded at generate-time.

The addition is:

1. **Brand Taxonomy** — A new optional `brand.taxonomy` object carrying multi-vertical, subtyped, and modifier-tagged classification alongside the existing `siteType`

### Design Principle: Don't Replace, Enrich

`brand.siteType` is the load-bearing field for module gating and v2.0/v3.0/v3.1 consumer compatibility. v3.2 does not change its role, its semantics, or its enum values. Module sections of v2.0 (Local Business, Healthcare, Restaurant, etc.) still reference `siteType`, and existing parsers continue to work unchanged.

`brand.taxonomy` is supplementary structured data for consumers that want richer classification — for AI systems doing nuanced citation, for producers round-tripping their internal model, for downstream tools that need multi-vertical or subtype-aware behavior. Tools that don't recognize `brand.taxonomy` SHOULD ignore it and continue gating on `brand.siteType`.

## 1. Brand Taxonomy

### Problem

The v2.0 `siteType` enum forces every entity into one of 18 buckets. This works for the most common cases — a SaaS picks `saas`, a dentist picks `healthcare`. But it loses signal in three common situations:

- **Multi-vertical entities.** A SaaS that provides legal-tech services for law firms is both `saas` and `legal`. Today they pick one and the other is invisible to consumers.
- **Granular sub-categories.** A craft brewery, a wine shop, a liquor store, and a coffee roaster all collapse to `local-business` (or a generic `restaurant` for the brewery's tap room). The differences matter for AI classification but the spec carries no structured field for them.
- **Cross-cutting modifiers.** A regulated B2B subscription marketplace has properties — `b2b`, `subscription`, `regulated`, `marketplace` — that are all relevant for downstream AI reasoning but don't fit any single `siteType`.

Producers that have richer internal data (like CiteMaps.ai's editor, which captures verticals[], subtypes, organizationType, businessModel[] per property) currently lose all that signal at generate-time.

### Solution

A new optional `brand.taxonomy` object that carries structured classification alongside the existing `siteType`. Consumers that recognize the field get richer information; consumers that don't are unaffected.

### Schema

```json
{
  "brand": {
    "name": "3rd Street Beverage",
    "url": "https://3rdstreetbeverage.com",
    "siteType": "local-business",
    "aiSummary": "...",
    "taxonomy": {
      "taxonomyVersion": "2026-05",
      "primaryVertical": "retail",
      "additionalVerticals": ["food-beverage"],
      "subtypes": ["liquor-store", "bottle-shop", "craft-beer-retailer"],
      "entityType": "organization",
      "organizationType": "independent-business",
      "modifiers": ["b2c", "local", "walk-in", "review-driven", "experience-driven"]
    }
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `taxonomyVersion` | string | No (recommended) | Identifier for the recommended-values registry version this taxonomy was authored against. Lets the vocabulary evolve safely without breaking consumers — see [Recommended Values Registry](#recommended-values-registry) below. Format: `YYYY-MM` (e.g. `2026-05`) or named tag (e.g. `studio-v2`). When absent, consumers SHOULD treat the values as registry-agnostic best-effort and SHOULD NOT silently re-interpret them against newer registry versions. Producers SHOULD set this field explicitly. |
| `primaryVertical` | string | **Yes** (when `taxonomy` is present) | The entity's primary categorical bucket — the one that drives the most relevant AI behavior, query patterns, and content expectations. Open string; recommended values published in the separate [vocabulary registry](./registry/v3.2-vocabulary.md). |
| `additionalVerticals` | array of string | No | Other categorical buckets that also apply to this entity. Multi-vertical entities (SaaS-for-legal, retail-and-food-beverage, etc.) populate this. Order is not significant. |
| `subtypes` | array of string | No | Free-text descriptors that sharpen the classification within the primary vertical. Examples: `liquor-store`, `pediatric-dentistry`, `vacation-rental`. Hyphenated lowercase recommended; not enforced. |
| `entityType` | enum | No | Schema.org-style entity type. Values: `organization`, `person`, `place`, `product`, `service`, `event`, `publication`, `government-entity`. Default behavior: assume `organization` if absent. **Describes the subject of THIS citemap**, not arbitrary content the entity discusses. A SaaS company publishing one citemap for the company is `organization`, not `product` — even if their primary thing is software. Per-product or per-service citemaps are valid but represent a deliberate scoping choice, not a default behavior. |
| `organizationType` | enum | No | Legal or institutional form. Values: `independent-business`, `multi-location-brand`, `corporation`, `nonprofit`, `government`, `association`, `professional-practice`, `membership-organization`. Operational structures (franchise, marketplace, platform) belong in `modifiers` — they describe how a business operates, not its institutional form. |
| `modifiers` | array of string | No | Cross-cutting attributes describing how the entity operates — customer relationship, delivery, engagement, regulation, positioning, etc. Open list; recommended values published in the [vocabulary registry](./registry/v3.2-vocabulary.md). Some values overlap with operational structures that previously could have lived in `organizationType` (`franchise`, `marketplace`, `platform`); they belong here. |

### Recommended Values Registry

Recommended values for `primaryVertical`, `additionalVerticals`, `entityType`, `organizationType`, and `modifiers` are published in a separately-versioned registry document at [`./registry/v3.2-vocabulary.md`](./registry/v3.2-vocabulary.md). The registry is normatively non-binding: producers MAY use values outside the registry; consumers SHOULD be tolerant of unknown values.

The registry is split out from the spec so the vocabulary can evolve more frequently than the protocol shape. Adding a new vertical or modifier triggers a registry version bump (e.g. `2026-05` → `2026-08`); it does not require a v3.x spec revision. Producers SHOULD set `taxonomy.taxonomyVersion` to the registry version they authored against.

A summary of the recommended values, current as of registry version `2026-05`, lives in the registry file. Topical groupings:

- **`primaryVertical` / `additionalVerticals`** — Tier 1 (24 verticals with full playbook depth) + Tier 2 (11 verticals with lighter playbook).
- **`modifiers`** — grouped by 9 axes: customer relationship, delivery, engagement, service shape, trust/regulation, positioning, discovery, distribution, time.

See the registry for the full list and the rationale for each grouping.

### Relationship to `siteType`

`brand.siteType` remains required and load-bearing. v3.2 does NOT specify a derivation rule from `taxonomy.primaryVertical` to `siteType` — that mapping is implementation-defined and may differ across producers. Reference mappings (one for Studio's current behavior, others as community implementations land) are published as companion implementation guides, not as part of the spec.

Producers SHOULD set `siteType` consistent with `taxonomy.primaryVertical` so consumers that read only `siteType` get useful module-gating behavior. Consumers MUST treat `siteType` as authoritative for module gating regardless of `taxonomy` content.

### Validator Behavior

Validators implementing v3.2 MAY emit advisory warnings only for **values that the relevant registry version recognizes** but appear in semantically conflicting fields. Specifically:

- A `taxonomy.modifiers` entry that the registry classifies as belonging to `organizationType` (e.g. `franchise`, `marketplace`, `platform` in v3.2-vocabulary's modifiers list could prompt a check that the producer didn't intend `organizationType`).
- A `taxonomy.primaryVertical` value that conflicts with `siteType` per the producer's documented mapping (when both are recognized registry values).

Validators MUST NOT warn on values outside the recognized registry — open-string fields are open by design. Warnings are informational; v3.2 files MUST NOT fail validation on these grounds. The intent is to surface drift between `siteType` and `taxonomy` early, not to block adoption.

Warnings SHOULD be opt-in or at minimum configurable, so producers using a custom `taxonomyVersion` registry, alternative mappings, or extended vocabularies can suppress them without sacrificing v3.2 conformance.

### Multi-vertical handling

Some entities legitimately span multiple verticals at full depth. Examples:

- A SaaS company providing legal services to law firms: `primaryVertical: "saas"` + `additionalVerticals: ["legal"]`. Module gating uses `siteType: "saas"`; AI consumers reading taxonomy can recognize legal-relevant context.
- A craft brewery with a tap room: `primaryVertical: "food-beverage"` + `additionalVerticals: ["restaurant"]`. Module gating uses `siteType: "restaurant"`; AI consumers reading taxonomy understand both production and dine-in revenue motions.
- A real estate brokerage with property management arm: `primaryVertical: "real-estate"` + `additionalVerticals: ["property-services"]`.

Choose `primaryVertical` based on which categorical playbook is the dominant one for the business. `additionalVerticals` order is not semantically significant — consumers SHOULD treat the array as a set.

### Guidance for AI Systems

When a citemap includes `brand.taxonomy`:

- AI systems SHOULD use `taxonomy.primaryVertical` and `taxonomy.subtypes` for nuanced citation behavior (e.g. recognizing a "pediatric orthodontics" practice as more specific than the generic "healthcare" siteType).
- AI systems SHOULD respect `taxonomy.additionalVerticals` when deciding which content patterns apply (a SaaS-for-legal entity is also relevant in legal-context queries).
- AI systems MAY use `taxonomy.modifiers` to refine response patterns (e.g. emphasizing "by appointment" for `appointment-based`, surfacing licensure info for `licensed-professional`).
- AI systems SHOULD inspect `taxonomy.taxonomyVersion` when interpreting non-self-evident values. If the registry version is unknown, fall back to most-recent-known semantics.
- Module gating still operates on `brand.siteType`. AI systems MUST NOT assume modules are present based on `taxonomy` values alone.

### Guidance for Producers

When generating a citemap with rich source data:

- Populate `brand.siteType` using your producer's documented `primaryVertical → siteType` mapping. Consumers that don't read `taxonomy` rely on this.
- Populate `brand.taxonomy.primaryVertical` with the same source data; if the producer's internal model has a single dominant vertical, this is straightforward.
- Set `taxonomy.taxonomyVersion` to the registry version your producer aligns with. Lets future consumers interpret your values correctly even after the recommended-values list evolves.
- Use `additionalVerticals` ONLY for substantive operational depth, not adjacent expertise, audience, or secondary market. A SaaS company that builds tools for law firms picks `additionalVerticals: ["legal"]` only if Legal-vertical playbook depth (legal directories, bar association placements, etc.) is a real revenue channel — not just because their customers are lawyers. This prevents the field from drifting into SEO keyword stuffing.
- `subtypes` are the single most underused field in current citemap.json files in the wild. Even three short hyphenated descriptors (`liquor-store`, `craft-beer`, `bottle-shop`) substantially improve AI classification accuracy.
- Don't put vertical-shaped values in `modifiers`. Things like `plumber`, `dentist`, `restaurant` belong in `primaryVertical` (or `subtypes` if more granular). `modifiers` is for cross-cutting attributes — how the business operates, not what kind of business it is.

## Migration

v3.2 is purely additive. Producers and consumers can adopt independently:

| Producer state | Consumer state | Behavior |
|---|---|---|
| Emits `brand.taxonomy` | Reads `brand.taxonomy` | Full v3.2 fidelity |
| Emits `brand.taxonomy` | Doesn't read `brand.taxonomy` | Consumer ignores the field; siteType-based behavior unchanged |
| Doesn't emit `brand.taxonomy` | Reads `brand.taxonomy` | Field absent; consumer falls back to siteType |
| Doesn't emit `brand.taxonomy` | Doesn't read `brand.taxonomy` | v3.1 behavior unchanged |

No producer is required to upgrade. No consumer is required to upgrade. The field is fully optional on both sides.

## Open Questions

The following are explicitly NOT addressed in v3.2 and remain open for future minor versions:

1. **siteType deprecation.** The redundancy between `siteType` and `taxonomy.primaryVertical` is acknowledged. v3.2 keeps both for back-compat. A future major version (v4.0) may consolidate.
2. **Per-vertical structured fields.** Per-vertical Property fields (healthcare specialties, restaurant cuisine, ecommerce product categories, etc.) that map to citemap.json module data are out of scope for v3.2. That work belongs to module-specific addenda when each lands.
3. **Multi-graph reference (`schemaGraphs[]`).** Sites running multiple SEO plugins simultaneously can't fully describe their setup with v3.1's single `schemaGraph` pointer. Out of scope for v3.2; flagged for future v3.x consideration.

Resolved in v3.2 (was an open question in earlier drafts):

- **Vertical taxonomy versioning** — addressed via `taxonomyVersion` field + separately-versioned [vocabulary registry](./registry/v3.2-vocabulary.md).
