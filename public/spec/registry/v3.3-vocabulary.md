# citemap.json v3.3 Recommended Values Registry

> Vocabulary for `brand.taxonomy`, `@graph` / `relationships[]` edge types, and the foundational-section catalog introduced in [v3.3](../v3.3.md). Versioned independently of the spec.

**Registry version:** 2026-06
**Spec version:** 3.3
**Status:** Living document
**License:** MIT

---

## Why a separate registry

The v3.3 spec defines the *shape* of `brand.taxonomy`, the `@graph` extension, and the foundational-section contract — field names, types, required/optional status, validator behavior. The *vocabulary* (which verticals exist, which edge types are recognized, which modifiers are common, which entity types exist) is a different artifact: it evolves more frequently, doesn't change the protocol, and shouldn't trigger spec version bumps.

This registry is referenced by v3.3's `brand.taxonomy.taxonomyVersion` field. Producers writing a citemap SHOULD set `taxonomyVersion` to the registry version they authored against (e.g. `"2026-06"`); consumers that recognize the version interpret values according to that registry's semantics.

Adding a new vertical, modifier, or edge type triggers a registry version bump. Renames require a deprecation cycle of at least one published version interval — the old name keeps working via aliases while consumers migrate. Removals are last-resort and documented in the [Deprecation Log](#deprecation-log) with timeline.

## What changed from v3.2-vocabulary

The v3.2 vocabulary (registry version `2026-05`) carried forward into v3.3 with the following deltas:

**`primaryVertical` deltas:**

- `legal` → `legal-services` (renamed for scope clarity; legacy emits via alias)
- `financial` → `financial-services` (symmetric scope clarity; legacy alias)
- `events` (Tier 2) → `events-venue` (Tier 1, promoted + renamed)
- `research` (Tier 2) → `science-research` (Tier 2, renamed for scope clarity; legacy alias)
- `manufacturer` (Tier 1) — **RETIRED** per [v3.3 §6](../v3.3.md#6-vertical-taxonomy-retirements)
- `membership-org` (Tier 2) — **RETIRED** per [v3.3 §6](../v3.3.md#6-vertical-taxonomy-retirements)
- `industrial-services` (Tier 2) — **RETIRED** per [v3.3 §6](../v3.3.md#6-vertical-taxonomy-retirements)
- `local-business` — was already deprecated in v3.2; v3.3 confirms retirement per [§6](../v3.3.md#6-vertical-taxonomy-retirements)

Net count: **23 Tier 1 + 8 Tier 2 = 31 recommended `primaryVertical` values** (down from v3.2's 24 + 11 = 35).

**`entityType` deltas:** none. Unchanged from v3.2.

**`organizationType` deltas:** none. The `membership-organization` value remains valid — it described an institutional form, not a vertical, and was never the retired entry. (The retired Tier 2 vertical `membership-org` is a different concept; see [Deprecation Log](#deprecation-log).)

**`modifiers` deltas:** none. Carry forward from v3.2.

**New: edge-type vocabulary.** 13 relationship types introduced for the v3.3 `@graph` extension. See [Edge type vocabulary](#edge-type-vocabulary).

**New: foundational-section catalog.** 10 always-on sections enumerated for v3.3's foundational-section contract. See [Foundational sections](#foundational-sections).

## `entityType`

Schema.org-style entity type. Identifies what kind of thing this citemap describes. Default: `organization`. Unchanged from v3.2.

| Value | Description |
|---|---|
| `organization` | A business, institution, or other organized entity |
| `person` | An individual professional, public figure, or named subject |
| `place` | A geographic location, landmark, or natural area |
| `product` | A specific named product (consumer goods, software, physical or digital) |
| `service` | A specific named service offering |
| `event` | A scheduled or recurring event |
| `publication` | A periodical, blog, podcast, journal, or other publishing entity |
| `government-entity` | A government agency, department, or public institution |

`entityType` describes the SUBJECT of a citemap, not a product the entity sells. See v3.2 vocabulary registry for worked examples of non-organization entity types.

## `organizationType`

Legal or institutional form. Captures *what kind of organization* the entity is, structurally. Operational structures (franchise, marketplace, platform) belong in `modifiers`, not here. Unchanged from v3.2.

| Value | Description |
|---|---|
| `independent-business` | Privately held single-entity business; not part of a corporation or chain |
| `multi-location-brand` | A brand operating multiple owned locations (corporate, not franchised) |
| `corporation` | A legally incorporated entity, public or private |
| `nonprofit` | A 501(c)(3) or equivalent charitable organization |
| `government` | A government agency, department, or public-sector entity |
| `association` | A professional or industry association |
| `professional-practice` | A licensed-professional entity (law firm, medical practice, accounting firm) |
| `membership-organization` | An organization whose primary structure is dues-paying members (chamber, alumni group, club) |

### Catchalls and retired entries

The following values are **RETIRED** in v3.3 — they remain valid strings for back-compat (validators MUST NOT reject them per v3.3 §6) but are no longer recommended as `primaryVertical`. Producers MUST migrate to a real vertical per the suggested replacements:

| Value | Status | Replacement |
|---|---|---|
| `local-business` | RETIRED (was already deprecated in v3.2; v3.3 confirms) | Pick a real specific vertical + `modifiers: ["local"]` + `Property.scope: "local"`. The siteType enum still includes `local-business` for module-gating back-compat. |
| `manufacturer` | RETIRED in v3.3 | Entity's actual vertical (`industrial-supplier`, `medical-devices`, `food-beverage`, etc.) + `modifiers: ["manufacturing-direct"]` |
| `membership-org` | RETIRED in v3.3 | Closest content vertical (`nonprofit`, `legal-services`, `science-research`, etc.) + `organizationType: "membership-organization"` |
| `industrial-services` | RETIRED in v3.3 | Real work vertical (`construction`, `logistics`, `professional-services`, etc.) + `modifiers: ["b2b"]` |

Validators SHOULD warn on retired values per [v3.3 §9.8](../v3.3.md#98-retired-primaryvertical-advisory) and suggest the migration target consistent with the entity's existing modifiers and organizationType.

## Edge type vocabulary

**New in v3.3.** Vocabulary for the `relationships[]` array in the v3.3 graph extension. Every edge type has a stable type-vocabulary URI at `https://citemaps.org/v3.3/types/{type}`. See [v3.3 §2](../v3.3.md#2-edge-taxonomy-v1) for the full architectural definition.

### Schema.org-native types (7) — mirrored inline + as relationship row

| Type | Schema.org property | Inverse | Use case |
|---|---|---|---|
| `parentOrganizationOf` | `subOrganization` | `subOrganizationOf` | "I am a parent of X" — emit on the parent's citemap |
| `subOrganizationOf` | `parentOrganization` | `parentOrganizationOf` | "I am a subsidiary of X" — emit on the subsidiary's citemap |
| `memberOf` | `memberOf` | `member` | "I am a member of X" — associations, industry groups, accreditation councils |
| `worksFor` | `worksFor` | `employee` | "Person → Organization" employment |
| `affiliatedWith` | `affiliation` | (no clean inverse) | Looser association than `worksFor` — advisory, board, fellowship |
| `founderOf` | `founder` | (no clean inverse) | "Person → Organization they founded" |
| `sameAs` | `sameAs` | `sameAs` (symmetric) | Same-entity identity claims — Wikipedia, Wikidata, Crunchbase, LinkedIn |

### CiteMaps-extension types (6) — relationship row only, no schema.org inline

| Type | Use case | Verification notes |
|---|---|---|
| `franchiseOf` | Franchisee → franchisor | Distinct from `subOrganizationOf`; franchises typically aren't legal subsidiaries but are AEO-relevant. Few franchisors publish full citemaps — most edges stay unverified. |
| `locationOf` | A specific physical location of a multi-location org | Locations typically aren't separate legal entities. Per-location citemaps verify reciprocally; catalog-entry locations stay unverified. |
| `practitionerAt` | Person → Organization (alternate to `worksFor`) | Healthcare / legal / professional-services context; clinical/professional role rather than employment. |
| `predecessorOf` / `successorOf` | Entity continuity through rebrand, acquisition, spin-off | Both directions emit as separate edges. Verification rare in practice since predecessor entities are typically dissolved. |
| `operatesAs` | Self-edge for multi-vertical roles within one entity | `operatesAs` self-edges (from = to) are intrinsically self-verifying; validators MAY set `verified: true` automatically. |
| `sellsAt` | Entity → Marketplace storefront | Pairs with `brand.storefronts[]`. Currently stays unverified — major marketplaces don't expose structured data confirming the inverse seller relationship. |

### Custom edge types

Producers MAY define custom edge types using absolute URIs at their own vocabulary namespace (e.g. `https://example.com/vocab/preferredVendor`). Validators MUST accept custom types but SHOULD warn (per v3.3 §2.6). When patterns emerge across multiple producers, custom types MAY graduate into this registry at a future version bump.

## Foundational sections

**New in v3.3.** The 10 always-on sections every v3.3 citemap.json emits regardless of vertical selection. See [v3.3 §3](../v3.3.md#3-foundational-sections-always-on-contract) for the full architectural definition.

| Section | Source | Status |
|---|---|---|
| `brand` | Producer's entity identity card | always-on in v3.2; unchanged |
| `citationContract` | Auto-generated per regen | always-on in v3.2; unchanged |
| `temporalRecord` | Producer's events/timeline data | empty array as explicit "no claims" signal |
| `trust.perField` | Computed from field provenance + NAP corroboration | always-on in v3.3 |
| `answerContent` | Producer's authored Q&A (or AI-suggested fallback) | always-on in v3.3 |
| `brand.team` | Producer's team roster | always-on in v3.3 |
| `brand.products` | Producer's product catalog | always-on in v3.3 |
| `brand.sameAs` | Verified social profiles + storefronts auto-fold | always-on in v3.3 |
| `policy` | Producer's editorial / correction / disclosure / privacy URLs | NEW in v3.3 |
| `verification` | Producer's external verifier claims | PROMOTED from optional module to foundational in v3.3 |

Producers MUST emit all 10 sections. Empty-emission is honest — `"team": []`, `"verification": { "claims": [] }` are explicit "no data" signals (per v3.3 §3.3). The single exception: `trust.perField` MAY be omitted if the producer has no provenance data; producers SHOULD emit at least `{}` to indicate section existence.

## Versioning Policy

- **Additive changes** (new vertical, new modifier, new entityType value, new edge type) trigger a registry minor version bump (e.g. `2026-06` → `2026-09`).
- **Renames** require a deprecation cycle. The old name keeps working as an alias for at least one published interval after the rename; aliases let consumers reading the old name keep parsing during the migration window.
- **Removals** are last-resort. A removed value stays documented in the [Deprecation Log](#deprecation-log) for at least 12 months after removal.
- **Spec-level changes** (e.g. adding a new field to `brand.taxonomy`, modifying the @graph shape) are NOT registry changes; they require a v3.x spec revision.

The registry version is what producers set via `taxonomy.taxonomyVersion`. A producer that wrote against `2026-05` (v3.2-vocabulary) continues producing valid v3.2 citemaps even after the registry advances to `2026-06`; the version tag tells future consumers what semantics to apply.
